Ceph migration

Your bytes. Not your client’s hostage.

A site’s data should never be captive to the one tool that happened to write it. BriX-Cache ships migration and rescue tooling that treats a RADOS cluster as your object store — not XrdCeph’s, not XrdCl’s. Read a stock striped-RADOS pool byte-for-byte, decode a CephFS with the MDS switched off, and move data between layouts on your terms — zero-move, verified, reversible.

The problem

Object layout is a lock-in you didn’t sign up for.

When a site commits to a storage client, it quietly commits to that client’s on-object format too — the stripe geometry, the OID naming, where checksums and namespace metadata are stashed. Years later the data is measured in petabytes and the format is a wall: you can’t change client without rewriting everything, and you can’t read it with anything else.

BriX-Cache refuses that bargain. It knows the stock XrdCeph striper layout and the CephFS on-disk format precisely enough to read either without conversion — so the format stops being a wall and becomes a starting point you can migrate off, into, or out of.

What you get back

Four freedoms, all citable.

Read stock XrdCeph, byte-for-byte

The flat RADOS backend probes whether an object is a stock libradosstriper object and, if so, reads it back through the exact upstream OID convention — first-stripe suffix, stripe-data detection, OID→PFN recovery, directory listing reconstructed over the flat keys. Existing petabytes stay readable without a rewrite.

Read a CephFS with no MDS

A read-only decoder walks CephFS on-disk metadata — backtrace and layout denc — straight out of RADOS. If the MDS is gone, the filesystem is still legible.

Move between layouts, both ways

Migrate bidirectionally between the XrdCeph striper layout, CephFS, flat RADOS, and a POSIX tree — zero-move by redirect, or intra-cluster server-side copy — with verify and rollback at every step.

Metadata travels with the bytes

Checksums-at-rest live as RADOS object xattrs (user.XrdCks.*) and are carried through every path, so a migrated object arrives with the same recorded digest it left with — integrity is not re-asserted, it is preserved.

The toolchain

Five tools in the brix-tools package.

All ship in the brix-tools RPM subpackage, with their migration guides installed as docs. Forward, reverse, flatten, and two offline rescuers for when the namespace service is already gone.

xrdceph_striper_migrate FORWARD

Enable CephFS over an existing stock-XrdCeph / libradosstriper RADOS pool. Zero-move by default (RADOS redirect stubs, no bytes copied); also --mode copy for server-side OSD→OSD, plus --finalize, --rollback, and --verify (adler32 against the carried user.XrdCks.adler32).

xrdceph_cephfs_to_striper REVERSE

The other direction: expose an unmounted CephFS as stock-XrdCeph (libradosstriper) storage, zero-move via redirects, then --finalize to a self-owned striper layout so the CephFS can be torn down.

xrdceph_migrate FLAT → TREE

Migrate a flat RADOS pool (object-key == logical path) into a real filesystem tree by copy-through-mount — point the destination at a CephFS mount and the MDS builds the namespace. Carries user.* xattrs including the checksums-at-rest.

xrdrados_rescue RESCUE

Offline, pure-RADOS recovery: ls / stat / get / cp objects straight out of a flat pool with no namespace service running at all.

xrdcephfs_rescue RESCUE

Offline CephFS recovery with no MDS, no mount, no libcephfs — reads a quiesced CephFS directly from its RADOS pools via a read-only decoder: ls / stat / cat / get / cp -r.

How zero-move works

A redirect first, a rewrite only if you say so.

Zero-move migration doesn’t copy bytes: it lays down RADOS redirect stubs so the new layout points at the existing objects. That makes the cut-over instant and cheap — but it is a staging state, not a finished one.

While staged, a redirect stub is write-through and delete-through to the source: deleting a stub deletes the original. It is safe only until --finalize makes the new layout the true owner. Rollback unsets the manifest before unlinking. And the one thing that is a firm no-go is a zero-copy upgrade of a flat pool to CephFS — the MDS has to mint real inodes, so that move always copies.

xrdceph_striper_migrate · zero-move
$ xrdceph_striper_migrate --pool data \
  --target cephfs # redirect stubs, no copy
$ xrdceph_striper_migrate --verify # adler32 vs xattr
# happy? make the new layout the owner:
$ xrdceph_striper_migrate --finalize
# or back out cleanly:
$ xrdceph_striper_migrate --rollback
The honest boundary

What this does — and doesn’t — cover.

CVMFS is a separate plane

BriX-Cache serves CVMFS as its own content-addressed protocol, alongside these tools — but the migration toolchain does not touch CVMFS object format. The freedom here is from the XrdCeph / XrdCl striper and CephFS layouts, not from CVMFS.

Production reads need one site rule

The striper read path is contract-complete, but touching a real RAL/Glasgow pool needs the site’s own lfn2pfn name-mapping rule. That last-mile rule was never supplied to us, so production enablement is a site step, not a claim we make for you.

Validated on single-node Reef

The mechanics are verified on Ceph Reef 18.2.4 — but on a one-OSD demo cluster, which proves object and metadata correctness, not multi-OSD failure, recovery, or scrub-under-load. The live lab is infra-constrained, not yet CI-wired.

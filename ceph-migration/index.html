Ceph migration

Move your data in either direction.

BriX-Cache gives your site a clear, supported path between XrdCeph and CephFS. Read a stock striped-RADOS pool byte-for-byte, decode CephFS with the MDS switched off, and move between layouts in either direction. Choose zero-move redirects or server-side copy, verify the result, and roll back while the migration is staged.

Your starting point

Keep the layout you have. Choose where to go next.

Storage clients bring an on-object format with them: stripe geometry, OID naming, checksums, and namespace metadata. Once a site holds petabytes in that format, changing layouts needs a migration path that understands every one of those details.

BriX-Cache understands both the stock XrdCeph striper layout and the CephFS on-disk format precisely enough to read either without conversion. Your current layout becomes a well-understood starting point, with supported paths into and out of both XrdCeph and CephFS.

CephFS # on-disk, MDS optional

What this enables

Four practical migration capabilities.

Read stock XrdCeph, byte-for-byte

The flat RADOS backend probes whether an object is a stock libradosstriper object and, if so, reads it back through the exact upstream OID convention — first-stripe suffix, stripe-data detection, OID→PFN recovery, directory listing reconstructed over the flat keys. Existing petabytes stay readable without a rewrite.

Read a CephFS with no MDS

A read-only decoder walks CephFS on-disk metadata — backtrace and layout denc — straight out of RADOS. If the MDS is gone, the filesystem is still legible.

Move between layouts, both ways

Migrate bidirectionally between the XrdCeph striper layout, CephFS, flat RADOS, and a POSIX tree — zero-move by redirect, or intra-cluster server-side copy — with verify and rollback at every step.

Metadata travels with the bytes

Checksums-at-rest live as RADOS object xattrs (user.XrdCks.*) and are carried through every path, so a migrated object arrives with the same recorded digest it left with — integrity is not re-asserted, it is preserved.

The toolchain

Five tools in the brix-tools package.

All ship in the brix-tools RPM subpackage, with their migration guides installed as docs. Move to CephFS, move to XrdCeph, convert a flat pool into a tree, or recover data offline when the namespace service is unavailable.

xrdceph_striper_migrate TO CEPHFS

Move from a stock XrdCeph / libradosstriper RADOS pool to CephFS. Choose zero-move RADOS redirects or --mode copy for server-side OSD→OSD transfer, then verify, complete, or roll back the migration with confidence.

xrdceph_cephfs_to_striper TO XRDCEPH

Move from an unmounted CephFS to stock XrdCeph (libradosstriper) storage. Start with zero-move redirects, verify the result, then complete the move to a self-owned striper layout when you are ready.

xrdceph_migrate FLAT → TREE

Migrate a flat RADOS pool (object-key == logical path) into a real filesystem tree by copy-through-mount — point the destination at a CephFS mount and the MDS builds the namespace. Carries user.* xattrs including the checksums-at-rest.

xrdrados_rescue RESCUE

Offline, pure-RADOS recovery: ls / stat / get / cp objects straight out of a flat pool with no namespace service running at all.

xrdcephfs_rescue RESCUE

Offline CephFS recovery with no MDS, no mount, no libcephfs — reads a quiesced CephFS directly from its RADOS pools via a read-only decoder: ls / stat / cat / get / cp -r.

How zero-move works

A redirect first, a rewrite only if you say so.

Zero-move migration doesn’t copy bytes: it lays down RADOS redirect stubs so the new layout points at the existing objects. That makes the transition quick and efficient while keeping the migration in a staged, reversible state.

While staged, a redirect stub is write-through and delete-through to the source: deleting a stub deletes the original. Use --rollback to move back while the migration is staged. When verification succeeds, --finalize completes the migration and makes the new layout the owner. Moving a flat pool to CephFS always copies because the MDS needs to create real inodes.

xrdceph_striper_migrate · zero-move
$ xrdceph_striper_migrate --pool data \
  --target cephfs # redirect stubs, no copy
$ xrdceph_striper_migrate --verify # adler32 vs xattr
# verification passed: complete the migration
$ xrdceph_striper_migrate --finalize
# while staged, move back cleanly instead
$ xrdceph_striper_migrate --rollback
Scope and validation

What is covered today.

CVMFS is a separate plane

BriX-Cache serves CVMFS as its own content-addressed protocol, alongside these tools — but the migration toolchain does not change the CVMFS object format. These tools focus on the XrdCeph striper and CephFS layouts.

Production reads need one site rule

The striper read path is contract-complete, but touching a real RAL/Glasgow pool needs the site’s own lfn2pfn name-mapping rule. Add that site-specific rule as the final production enablement step.

Validated on single-node Reef

The mechanics are verified on a one-OSD Ceph Reef 18.2.4 demo cluster, covering object and metadata correctness. Multi-OSD failure, recovery, and scrub-under-load validation remain future live-lab work.

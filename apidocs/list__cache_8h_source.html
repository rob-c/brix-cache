1/*
2 * list_cache.h — per-worker LRU cache of sorted ListObjects results (W6c).
3 *
4 * WHAT: caches the *sorted* (key + is_prefix) list produced by a full bucket
5 * walk, keyed by (root, prefix, delimiter), so paginating a large bucket page
6 * by page does not re-walk + re-sort the whole subtree on every request
7 * (turning the per-bucket listing cost from O(n^2) back into O(n) amortised).
8 *
9 * WHY (design): the cache is process-local (per nginx worker) rather than SHM.
10 * The shared KV store reserves a fixed key_max+val_max stride per slot, so a
11 * value large enough to hold a big sorted listing would waste that much memory
12 * on every slot and still cap the listing size — i.e. it cannot cache exactly
13 * the large buckets that benefit most. A heap LRU sizes each entry to its
14 * actual content and is not shared across workers (each warms its own; the
15 * listing is still O(n) amortised per worker).
16 *
17 * CONSISTENCY: a cached listing is reused only while the bucket-root directory
18 * mtime is unchanged AND within the configured TTL. Root mtime catches
19 * top-level mutations; the TTL bounds staleness from deeper-subtree changes.
20 * This is bounded eventual consistency (S3 ListObjects is historically
21 * eventually-consistent), and the feature is opt-in (default off) so
22 * strongly-consistent deployments leave it disabled.
23 */

1/* client/lib/vfs.h
2 *
3 * vfs.h — client storage-backend abstraction.
4 *
5 * WHAT: one open-file handle (pread/pwrite/fstat/truncate/sync/commit/abort/close)
6 * over a backend vtable, so a copy endpoint can be local POSIX, a block
7 * device, or an S3/object store interchangeably.
8 * WHY: copy.c hard-codes POSIX fds for the local endpoint and threads three
9 * different ad-hoc I/O seams (pump_src/sink_fn, iobuf be+pread/pwrite,
10 * brix_disk_ring). One handle unifies them and lets non-POSIX backends in.
11 * HOW: the keystone is commit(): POSIX finalises by fsync+rename(temp->final);
12 * block by fsync; S3 by multipart-complete. caps lets the copy engine pick
13 * a path (random-write vs append-only, atomic-temp vs native commit).
14 * ngx-free.
15 *
16 * BACKEND I/O: this VFS is the client-side *shell* (URL/scheme resolution,
17 * credential store, commit/abort, io_uring fast-path). The raw byte I/O of
18 * the POSIX and block backends is dispatched through the SHARED Storage
19 * Driver `brix_sd_posix_driver` (src/fs/backend/sd.h, ngx-free under
20 * -DXRDPROTO_NO_NGX via libxrdproto) — the SAME driver the nginx server's
21 * data plane uses (src/fs/vfs/vfs_io_core.c). So `pread`/`pwrite`/`fstat`/
22 * `ftruncate`/`fsync` have ONE implementation across client and server.
23 * (The old "never reuse sd.h — phase-55 broke it" caveat is obsolete: sd.h
24 * now carries a complete ngx-free fallback, so the worker-safe raw-fd
25 * surface is shared without pulling nginx into the client.)
26 */

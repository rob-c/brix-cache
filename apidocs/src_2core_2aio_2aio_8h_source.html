8/*
9 * AIO — async file I/O via the nginx thread pool, plus response builders.
10 *
11 * WHY AIO?
12 * nginx worker threads are single-threaded and must never block. Calling
13 * pread(2) directly would stall the event loop for the duration of the disk
14 * I/O, preventing any other connection from making progress. Instead we:
15 *
16 * 1. Allocate a task context struct (e.g. brix_read_aio_t) on the heap.
17 * 2. Post an ngx_thread_task_t to the nginx thread pool.
18 * 3. The thread-pool worker calls *_aio_thread() — pread/pwrite happens here.
19 * 4. When done, nginx posts the *_aio_done() callback back to the main thread.
20 * 5. The done callback queues the response and calls brix_aio_resume() to
21 * wake the connection.
22 *
23 * While AIO is in flight, ctx->state = XRD_ST_AIO and both read/write events
24 * are disarmed. ctx->destroyed is checked in every done callback so that a
25 * callback firing after client disconnect safely discards its work.
26 */
27

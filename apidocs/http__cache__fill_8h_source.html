4/*
5 * http_cache_fill.h - off-event-loop cache-miss fill for the HTTP read plane
6 * (phase-64 SP2 "shell -> full").
7 *
8 * WHAT: One helper, brix_http_cache_fill_if_needed(), that the WebDAV GET and
9 * S3 GetObject handlers call before their inline open+serve. When a read
10 * would be a cache MISS over a remote tier (a source the worker must reach
11 * over a socket, or a store it writes over the network), the helper runs
12 * the fill on the shared thread pool and suspends the request, re-entering
13 * the handler - now a cache HIT, served zero-copy - on completion.
14 *
15 * WHY: The cache decorator (sd_cache) fills a miss synchronously inside open().
16 * That is correct on a worker thread but a stall on the nginx event loop:
17 * a socket wire client (sd_xroot) cannot do blocking I/O on the un-pumped
18 * loop (it fails EIO), and an in-process store (rados) freezes the whole
19 * worker for the transfer. The HTTP data plane runs storage I/O on the
20 * loop, so the fill must be moved off it. This mirrors the established
21 * thread-task pattern already used by WebDAV COPY/MOVE/PUT and S3 PUT.
22 *
23 * HOW: brix_sd_cache_fill_needs_offload() answers (without blocking) whether
24 * the inline open would stall; if so the helper posts an
25 * brix_sd_cache_fill_key() task and takes r->main->count++, returning
26 * NGX_DONE. The completion event (on the loop) re-enters the handler on
27 * success or finalizes with 502 on a fill failure. A COMPLETE hit, a local
28 * source+store, a slice object, or a missing thread pool all return
29 * NGX_DECLINED so the caller proceeds with its normal inline path.
30 */

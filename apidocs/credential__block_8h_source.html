4/*
5 * credential_block.h — the reusable `brix_credential <name> { … }` config block
6 * (phase-63 §14).
7 *
8 * WHAT: A named, declared-once identity a server uses to reach an UPSTREAM (a
9 * source backend, the cache fill / write-back flush, or — later — the
10 * proxy/TPC upstream). Parses into a small POD, interned by name, looked up
11 * by `credential=<name>` (or the sibling `brix_storage_credential <name>`)
12 * on the consuming directive.
13 *
14 * WHY: Today the same X.509-proxy + CA + bearer-token vocabulary is spelled five
15 * different ways across the cache-origin, stream-proxy and WebDAV-proxy
16 * config families (§14.1). One block, referenced everywhere, replaces that
17 * fragmentation — "identity is a property of the source, not of each
18 * subsystem that reaches it".
19 *
20 * HOW: `brix_conf_credential_block()` is an nginx block handler (the ngx_map /
21 * ngx_geo pattern): it records the name then re-enters ngx_conf_parse with a
22 * per-line handler that fills one field per nested directive. The result is
23 * interned in a small per-process table. `brix_credential_lookup()` finds a
24 * block by name; `brix_credential_bearer()` resolves its bearer token
25 * (inline `token`, else the contents of `token_file`). The X.509 fields are
26 * parsed and stored for C-3 (in-process GSI) but not yet consumed here.
27 */

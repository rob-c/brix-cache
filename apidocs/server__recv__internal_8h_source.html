4/*
5 * cms/server_recv_internal.h — private cross-file contract for the CMS
6 * server-side receive path.
7 *
8 * WHAT: Declares the handful of entry points shared between the four
9 * translation units that together implement an accepted CMS data-server
10 * connection's receive path — the lifecycle/close helpers
11 * (server_recv_lifecycle.c), the wire payload parsers (server_recv_parse.c),
12 * the opcode frame handlers + dispatch table (server_recv_frame.c), and the
13 * read/write event loop (server_recv.c).
14 *
15 * WHY: server_recv.c was a single 1038-line file spanning four distinct
16 * concerns (Phase-79 file-size split). Splitting it into focused units keeps
17 * each file under the 500-line guideline and lets each concern be reviewed in
18 * isolation. The only cost is that a few functions that were `static` now
19 * cross a translation-unit boundary; those — and only those — are declared here
20 * so the definer and the caller agree on the prototype (a header-declared symbol
21 * MUST be non-static at its definition or the link fails).
22 *
23 * HOW: Include after "server.h" (which provides brix_cms_srv_ctx_t and the
24 * brix_sess_* types). Purely internal — never included outside this module.
25 *
26 * Requires: server.h before inclusion.
27 */

4/*
5 * stream_wmirror.h — Phase 24 W3: XRootD stream DATA-write mirroring.
6 *
7 * WHAT: Mirrors a client's open(write) -> write/pgwrite -> close sequence to an
8 * ISOLATED shadow XRootD server so a new storage backend's write path can be
9 * validated against real production writes. Unlike the stateless read/metadata
10 * mirror (stream_mirror.c), data writes are stateful: the client's file handle is
11 * meaningless on the shadow and the payload spans many frames.
12 *
13 * HOW (low-risk design): per write-open, the bytes are accumulated into a bounded
14 * per-file buffer attached to the client connection context. On kXR_close the
15 * buffered file is handed to a DETACHED replay (its own cycle-pool context, the
16 * same fire-and-forget lifetime as the read mirror) that bootstraps a fresh shadow
17 * session and performs open(create) -> write -> close. This avoids a persistent
18 * client-lifetime shadow connection (and the use-after-free hazards that come with
19 * it). A write that exceeds the per-file or per-connection byte cap, or is sparse
20 * / non-sequential, aborts that file's mirror (counted) and never blocks the
21 * client — write mirroring is best-effort validation.
22 *
23 * GATING: no-op unless brix_mirror_writes is on, OP_WRITE is selected in
24 * brix_mirror_opcodes, and a stream mirror target is configured. The target
25 * MUST be an isolated namespace; replaying writes onto the primary's store would
26 * corrupt it.
27 *
28 * Requires: types/context.h (brix_ctx_t), protocol headers, nginx headers.
29 */
30

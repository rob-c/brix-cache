4/*
5 * WHAT: Translate upstream XRootD responses back to the client — kXR_redirect, kXR_wait, kXR_ok, and
6 * kXR_error handling with header reconstruction and state transitions.
7 * WHY: The proxy relays opaque wire frames between client and backend server. Upstream responses must be
8 * re-wrapped with a fresh ServerResponseHdr using the client's streamid (not upstream's), then queued
9 * for delivery to the client connection. Special cases redirect (cleanup upstream, resume client read),
10 * wait (schedule retry timer), and error (extract errno+message). State machine transitions ensure
11 * correct sequencing across proxy lifecycle phases.
12 * HOW: Single function brix_upstream_forward_response() with switch(status) covering all four response types.
13 * kXR_redirect rebuilds header + body, cleans upstream, queues to client. kXR_wait extracts seconds from
14 * body, sets timer handler, re-arms read event (epoll ET mode edge recovery). kXR_ok rebuilds header +
15 * body, cleans upstream, queues. kXR_error extracts errno/code/message, sends error frame. Default case
16 * aborts on unexpected status.
17 */
18

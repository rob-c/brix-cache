4/*
5 * cms/blacklist_file.h — file-driven server blacklist (Phase-89 W6′).
6 *
7 * WHAT: the manager re-reads an operator-maintained blacklist file (one entry
8 * per line: `host`, `host:port`, or IPv4 `a.b.c.d/n` CIDR) whenever its mtime
9 * changes, and on every poll re-asserts a registry blacklist against each
10 * currently-registered data server that matches an entry.
11 *
12 * WHY: cmsd parity (phase-61 W6′) — operators expect a cms.blacklist-style
13 * file that wins over transient state. Re-asserting every poll (rather than
14 * only on mtime change) is what makes the file authoritative: an admin-API
15 * `undrain` and the blacklist-clear inside brix_srv_register() are both
16 * re-covered within one poll period. Removing a line lifts the ban once the
17 * last re-asserted duration expires (3x the poll interval).
18 *
19 * HOW: state lives in the owning CMS-server srv conf (one per srv block, no
20 * globals). brix_cms_blfile_poll() is called from the per-connection ping
21 * tick and immediately after each node registration; it self-rate-limits the
22 * stat() to once per second, so per-node ping fan-in stays cheap. Matching
23 * is exact host text (covers IPv6 literals), optional exact port, or IPv4
24 * CIDR against the registry snapshot. Malformed lines are logged and
25 * skipped — a bad file never crashes the manager or drops the good lines.
26 */
27

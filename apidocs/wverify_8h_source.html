1/*
2 * wverify.h — self-computed write-integrity accumulator (backend-agnostic).
3 *
4 * WHAT: Accumulate a CRC-32 over the bytes a protocol WRITES to a storage
5 * handle, so that after close the object can be re-read through the
6 * storage driver and compared: proof the driver actually persisted every
7 * byte, rather than trusting the write path (the class of failure where a
8 * fd-keyed helper bypassed an object backend's block routing and the
9 * object read back short/empty).
10 * WHY: An object backend (pblock, rados) has no single kernel fd whose bytes
11 * are the file; the only trustworthy end-to-end check is "read it back
12 * through the same driver and confirm the content". This kernel supplies
13 * the write-side expectation for that compare, with no client cooperation.
14 * HOW: Each written extent [off, off+len) is CRC-32'd (zlib) and folded into an
15 * offset-sorted list; adjacent extents are coalesced via crc32_combine, so
16 * an in-order stream collapses to a single whole-file CRC and an
17 * out-of-order writer (GridFTP MODE E) still converges to the same value
18 * regardless of arrival order. brix_wverify_expected() yields the whole-
19 * object CRC only when the extents cover exactly [0, total) with no gap or
20 * overlap — a gap/overlap is itself a write-integrity failure.
21 *
22 * ngx-free (libc + zlib, malloc-owned) so it links into the shared compat kernel
23 * and is unit-testable standalone; the VFS seam (brix_vfs_wverify_check) drives
24 * the read-back compare through brix_cksum_u32_obj.
25 */

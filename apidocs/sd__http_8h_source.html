4/*
5 * sd_http.h — read-only HTTP(S) source storage driver (phase-63 C-4).
6 *
7 * WHAT: A capability-typed SD driver (CAP_RANGE_READ only; every write/dir/xattr
8 * slot NULL) that serves open/stat/pread/fstat/close against a plain
9 * HTTP(S) endpoint — `open` HEADs the URL for the size, `pread` issues a
10 * byte-Range GET. The sibling of the S3 `sd_remote` driver minus SigV4: it
11 * is the *generic* web source for the composable stack, so an export
12 * (`brix_storage_backend http://host:port/base`) or a read cache can reach
13 * any HTTP origin through the SAME driver→driver fill path C-1 uses for
14 * root:// — no bespoke per-scheme libcurl in the cache.
15 *
16 * HOW: The actual HTTP is performed by an INJECTED transport vtable
17 * (`brix_s3_transport_t`, the shared one the S3 driver uses) — the module
18 * passes the server libcurl singleton (`brix_s3_origin_curl_transport`),
19 * so this driver carries no libcurl dependency itself. No kernel fd ⇒ reads
20 * are memory-served. Blocking; runs on the cache-fill / AIO worker thread.
21 * Instances + objects are malloc-owned (no nginx pool), worker-safe.
22 *
23 * AUTH: anonymous GET by default. When `bearer_token` is set (the §14 credential
24 * block, threaded through the registry), every HEAD/GET carries
25 * `Authorization: Bearer <token>` — the WLCG / SciTokens path. Per-open
26 * credentials (open_cred) additionally present the requesting USER: a WLCG
27 * bearer as the Authorization header (phase-2 T7), or an x509 proxy PEM as
28 * a mutual-TLS client certificate on the origin handshake (phase-70 §5.1
29 * GSI-over-https) when the injected transport implements request_cred.
30 */

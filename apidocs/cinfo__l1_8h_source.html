4/*
5 * cinfo_l1.h - the per-worker write-through cinfo cache (phase-64 section 6.4).
6 *
7 * WHAT: A bounded per-worker LRU keyed by cache-object key -> the cinfo HEADER
8 * record (brix_cache_cinfo_t validity/stats/digest, no bitmap). It is the
9 * control-plane fast path the cstore reads cinfo through, so a warm hit on a
10 * REMOTE cache store costs no store round-trip and a fill batches one cinfo
11 * write on commit instead of one per block (section 6.4).
12 *
13 * WHY: When the cache store is a remote driver (roots:// / s3:// / a WebDAV
14 * server, SP2), reading the per-object cinfo via getxattr on every open
15 * would add a round-trip to each request. The L1 turns that into a local
16 * hash lookup; misses fall through to the store, and a fill mutates the L1
17 * header then writes it through once. For a LOCAL (posix) store the L1 is a
18 * cheap accelerator over the on-disk .cinfo and stays optional.
19 *
20 * HOW: A small malloc-owned hash table (FNV-1a over the key) with an intrusive
21 * MRU-at-head doubly-linked LRU list; inserting past the bound evicts the
22 * tail. Malloc-owned (no nginx pool) so it is safe to touch from the cache
23 * fill worker thread, exactly like the sd_cache / cstore state it backs. One
24 * instance per cstore per worker. See docs/refactor/phase-64-fully-tiered-
25 * composable-storage.md (section 6.4, Appendix I).
26 */

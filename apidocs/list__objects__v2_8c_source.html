6/*
7 * s3_handle_list — ListObjectsV2 XML response builder
8 *
9 * WHAT: Builds the ListBucketResult XML for S3 ListObjectsV2 requests. Parses query
10 * parameters (prefix, delimiter, continuation-token, max-keys), walks the filesystem
11 * to collect matching entries, applies pagination via continuation tokens, and emits
12 * a compliant XML response with Contents (objects) and CommonPrefixes (directories).
13 *
14 * WHY: S3 clients paginate list results using continuation tokens — each token encodes
15 * the last key returned in base64url. The server must skip all entries <= that key on
16 * subsequent requests. Delimiter-based common prefixes group keys under shared prefixes
17 * into <CommonPrefixes> elements per the S3 spec. XML entities (quotes, ampersands) can
18 * expand up to 6× in worst case, so buffer sizing uses 6× as a safety multiplier.
19 *
20 * HOW:
21 * 1. Parse query parameters: prefix, delimiter, continuation-token, max-keys,
22 * fetch-owner, encoding-type — all via s3_get_arg() with bounded buffers
23 * 2. Decode continuation token (b64url → start_after) to find pagination offset
24 * 3. Walk filesystem via s3_walk() collecting entries filtered by prefix/delimiter
25 * 4. Sort entries lexicographically via qsort(entry_cmp)
26 * 5. Skip entries <= start_after to apply continuation token (linear scan on sorted)
27 * 6. Slice [start_idx, end_idx) capped at max_keys — set truncated flag
28 * 7. Encode next continuation token from last returned key if truncated
29 * 8. Estimate XML buffer capacity (512 header + per-key × 6× expansion)
30 * 9. Append XML: ListBucketResult root → Name, Prefix, KeyCount, MaxKeys,
31 * IsTruncated, NextContinuationToken → Contents/CommonPrefixes loop
32 * 10. Create response buffer, increment metrics, send via brix_http_send_xml_buffer()
33 *
34 * Pool allocation: entries array and xml buffer use ngx_palloc(r->pool) — lifetime
35 * survives until the HTTP response filter completes.
36 */

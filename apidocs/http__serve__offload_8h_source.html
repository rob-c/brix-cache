4/*
5 * http_serve_offload.h - off-event-loop serve of a remote (socket-wire) object
6 * (phase-64 SP3: the serve-readback complement to the off-loop cache FILL).
7 *
8 * WHAT: One helper, brix_http_serve_offload_remote(), that the WebDAV GET and S3
9 * GetObject handlers call BEFORE the inline brix_vfs_open when the export's
10 * serve path reads from a socket-wire backend (xroot) - a primary root://
11 * backend, or a cache_store / stage_store served from one. It runs the WHOLE
12 * read - driver open (eager connect), cinfo load, miss fill, and the byte
13 * read - on the thread pool, materialising the object into a local temp file,
14 * then serves THAT through the normal sendfile pipeline.
15 *
16 * WHY: A socket wire client (sd_xroot) cannot do blocking I/O on the un-pumped
17 * event loop: not just the byte read (file_serve.c memory-backed serve) but
18 * even the open (sd_xroot_open eagerly connects + stats) and the cinfo load
19 * (kXR_fattr on a remote cache_store) fail there. So the entire serve, from
20 * open onward, must move off the loop - the read-only complement to the
21 * off-loop FILL that already moved cache-miss writes off the loop. In-process
22 * (rados) and curl (s3/http) backends block-but-complete on-loop and are
23 * served inline as before.
24 *
25 * HOW: A single thread task opens the composed instance for `key` (a cache open
26 * runs cinfo/fill/serve; a bare backend just opens), copies the object into an
27 * O_TMPFILE temp, and closes it. The completion adopts the temp fd as a POSIX
28 * VFS handle and calls brix_http_serve_file_ranged on it (zero-copy
29 * sendfile, owning range/headers/backpressure/lifecycle), runs the caller's
30 * protocol metrics callback, and finalises. A non-socket serve, a HEAD, or no
31 * thread pool returns NGX_DECLINED so the caller serves inline as before.
32 *
33 * PHASE-2 FOLLOW-UP (per-user backend credential on the offload path): `vctx`
34 * is the caller's ALREADY-BOUND VFS ctx (brix_vfs_ctx_bind_backend_cred
35 * already called on it, exactly as the caller's own inline open would
36 * need). brix_http_serve_offload_remote() runs the SAME credential gate
37 * (brix_vfs_backend_cred) on the event loop, BEFORE submitting the
38 * thread job — deny mode refuses (NGX_ERROR, errno EACCES) without ever
39 * opening the object. On grant it copies the resolved credential's
40 * strings into fixed buffers on the task ctx (the borrowed vctx/pool do
41 * not outlive this call the way the task does) and the worker thread
42 * rebuilds a stack brix_sd_cred_t from them to open via
43 * brix_sd_open_maybe_cred instead of the plain driver open.
44 */

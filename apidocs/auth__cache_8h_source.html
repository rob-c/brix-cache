1/*
2 * auth_cache.h — cross-worker cache of the combined auth-gate decision.
3 *
4 * brix_auth_gate() runs three scans per request (authdb rules, VO ACL,
5 * token scope). For hot data paths a pilot repeatedly hits the same
6 * {operation, path, identity} tuple and gets the same answer every time.
7 * This caches the boolean grant/deny so the repeated scans are skipped.
8 *
9 * The cache key (built in auth_gate.c) folds in every input to the decision —
10 * auth_level, need_write, resolved path, request path, DN, VO list and the
11 * raw token scope claim — so two requests only share a cache entry when they
12 * would genuinely reach the same verdict. TTL is short (default 30 s) so a
13 * config reload (which zeroes the zone) and any rule change converge quickly.
14 */

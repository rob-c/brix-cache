4/*
5 * sd_remote.h — remote-origin (s3://) storage driver: ranged reads plus staged
6 * whole-object writes.
7 *
8 * WHAT: A capability-typed SD driver (CAP_RANGE_READ|CAP_MEMFILE) serving
9 * open/stat/pread/preadv/fstat/close against a REMOTE object store, plus
10 * staged whole-object writes (.staged_* → single PUT or multipart upload)
11 * and .unlink (DELETE). The read-through cache fill drives it
12 * driver→driver: open the origin object, pread sequential ranges into the
13 * cache's staged-write sink; as a primary it also accepts uploads.
14 *
15 * WHY: It folds the origin onto the SAME SD seam as the local cache and
16 * export, so fills and uploads are origin-agnostic. There is deliberately
17 * no .pwrite slot — random in-place writes are impossible on an object
18 * store, so they are rejected at the cap layer while sequential uploads
19 * go through the staged path. Dir/xattr/namespace slots stay NULL.
20 *
21 * HOW: s3:// delegates to the shared S3 driver (sd_s3): the per-open object wraps
22 * an sd_s3_file*, pread is a signed Range GET. The HTTP transport is INJECTED
23 * by the cache (kept transport-agnostic, exactly like sd_s3), so no cache or
24 * libcurl dependency leaks into the backend layer. Blocking ops run only on
25 * the cache-fill worker thread. The instance + objects are malloc-owned
26 * (no nginx pool), so they are safe to build and use off the event loop.
27 */

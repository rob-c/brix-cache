1/*
2 * kv_internal.h — private on-slab layout types shared between the two halves of
3 * the SHM key/value store after the phase-79 file-size split.
4 *
5 * WHAT: Defines the two private structs that describe the slab-backed table
6 * layout — the per-zone header (lock + counters + geometry) and a single
7 * hash entry — used by both kv.c (data-plane ops: get/set/delete/stats)
8 * and kv_config.c (configuration + registry + zone init).
9 * WHY: kv.c (577 lines) exceeded the 500-line cap. The data plane and the
10 * configuration/registry plane are cohesive but independent — the only
11 * thing they share is this private table layout, so it is factored here
12 * rather than duplicated. These types are deliberately NOT in the public
13 * kv.h: consumers see only the opaque brix_kv_t handle, never the layout.
14 * HOW: Both translation units include this header. The header carries only the
15 * two struct typedefs; there are no cross-file function calls between the
16 * two halves, so no function crosses the boundary. Requires ngx core
17 * types (ngx_shmtx_sh_t, ngx_msec_t, fixed-width ints) before inclusion.
18 *
19 * Slab block layout (allocated FROM the zone's slab pool via
20 * brix_shm_table_alloc and published through shm_zone->data — it does NOT
21 * overlay shm.addr, which holds nginx's ngx_slab_pool_t header):
22 *
23 * +----------------------+ table base (shm_zone->data)
24 * | brix_kv_header_t | (lock must be first — ngx_shmtx_create target)
25 * +----------------------+ offset sizeof(header)
26 * | entry[0] | stride = sizeof(entry) + key_max + val_max
27 * | entry[1] |
28 * | ... |
29 * | entry[capacity-1] |
30 * +----------------------+
31 *
32 * capacity is the largest power of two that fits the configured zone size, so
33 * (hash & (capacity-1)) selects the home bucket without a modulo.
34 */

8 * WHAT: Defines all compile-time tunable constants for nginx-xrootd: read sizing (BRIX_READ_MAX 4 MiB per-vector element cap for normalising client readv requests; BRIX_READ_CHUNK_MAX 16 MiB wire chunk size splitting large contiguous reads into fewer sendfile boundaries; BRIX_RETURN_ERR combines access log + OpErr + send_error for error paths with code/msg).
9 *
10 * WHY: Compile-time constants prevent runtime allocation growth from unbounded client requests. Read sizing balances wire efficiency (16 MiB chunks reduce sendfile boundaries) against per-vector element cap (4 MiB normalises readv). File limit 16 prevents excessive fd consumption per connection. Walk depth 32 rejects deep symlink chains before expensive syscalls. Pool byte cap 64 MB allows sustained dirlist calls without heap exhaustion — connection closed with kXR_NoMemory on breach. Write payload 16 MiB handles non-default xrdcp chunk sizes with CRC overhead headroom. Auth attempts 10 protects against brute-force while allowing legitimate retry cycles (certreq + cert = 2 rounds per attempt). Clock skew 30 seconds accommodates NTP drift per WLCG recommendation. Send chain spin 16 keeps large sendfile responses moving without starving other ready connections via nginx posted-event queue yield. Metric macros conditional on ctx->metrics pointer — no overhead when metrics disabled. Return macros collapse common three-line pattern (log + metric + send) into single call for no-body success/error paths; handlers returning bodies keep explicit lines.

22/*
23 * Read sizing.
24 *
25 * BRIX_READ_MAX is the per-vector element cap used when normalising
26 * client readv requests. Large contiguous reads may still return up to
27 * BRIX_READ_REQUEST_MAX, but the response is split into larger 16 MiB
28 * wire chunks so cleartext sendfile responses need fewer header/file
29 * boundaries and therefore fewer writev/sendfile calls.
30 */
31#define BRIX_READ_MAX (4 * 1024 * 1024)
32#define BRIX_READ_CHUNK_MAX (16 * 1024 * 1024)
33#define BRIX_READ_REQUEST_MAX (64 * 1024 * 1024)
34
35/*
36 * Memory-budget streaming (Phase 31).
37 *
38 * BRIX_READ_WINDOW caps the *resident heap* a single in-flight read may hold
39 * at once, independently of the logical request size. A 64 MiB TLS read (which
40 * cannot use sendfile) is served as a sequence of fill -> encrypt -> drain
41 * cycles over a buffer of at most this size, instead of buffering the whole
42 * request in heap. The wire framing (BRIX_READ_CHUNK_MAX chunks) is unchanged
43 * — only the in-memory slice shrinks.
44 *
45 * BRIX_SCRATCH_TRIM_THRESHOLD is the high-water mark above which a per-session
46 * reusable scratch buffer is shrunk back to BRIX_READ_WINDOW once a request
47 * has fully drained. Hysteresis (2x window) avoids realloc thrash on sessions
48 * that legitimately oscillate around the window size.
49 *
50 * BRIX_CONN_XFER_HEAP_MAX bounds the combined size of one connection's
51 * transfer scratch buffers (read_scratch + read_hdr_scratch + write_scratch +
52 * payload_buf). With the read/write windows in place this ceiling is normally
53 * never reached; it is the per-connection backstop that turns a runaway buffer
54 * into a clean kXR_NoMemory rather than unbounded growth.
55 */
56#define BRIX_READ_WINDOW (2 * 1024 * 1024)
57#define BRIX_SCRATCH_TRIM_THRESHOLD (2 * BRIX_READ_WINDOW)
58#define BRIX_CONN_XFER_HEAP_MAX (4 * BRIX_READ_WINDOW)

60/*
61 * Optional io_uring disk-I/O backend (Phase 44).
62 *
63 * The backend is a third AIO dispatch tier (io_uring -> thread pool -> inline
64 * sync) selected per server block by `brix_io_uring on|off|auto`. These are
65 * compile-time, header-only constants; the runtime verdict is decided by the
66 * authoritative opcode probe in src/aio/uring.c, never by parsing `uname`.
67 *
68 * Mode enum (stored in ngx_stream_brix_srv_conf_t.io_uring, set via an
69 * ngx_conf_enum_t slot):
70 * OFF = never use io_uring (thread pool / inline only)
71 * ON = require io_uring — startup FAILS if it is not compiled in or the
72 * runtime probe fails (see §32 fail-fast; brix_uring_validate_conf)
73 * AUTO = enable iff the probe passes this process, else silent fallback
74 *
75 * QUEUE_DEPTH is the per-worker ring's SQ/CQ entry count = the ceiling on
76 * concurrent in-flight SQEs. Each read submits one SQE (windowed reads stream
77 * as one contiguous IORING_OP_READ per wire chunk), so depth tracks connection
78 * concurrency, not request size; get_sqe -> NULL simply falls back to the pool.
79 *
80 * The MIN_KERNEL_* values are a fast pre-filter only (5.6 = reliable
81 * register_eventfd for the completion bridge); RESTRICT_KERNEL_MINOR (5.10) is
82 * where io_uring_register_restrictions() becomes available (best-effort).
83 */
84#define BRIX_IO_URING_OFF 0
85#define BRIX_IO_URING_ON 1
86#define BRIX_IO_URING_AUTO 2

93/*
94 * Output-queue depth (Phase 29 pipelining; runtime-configurable).
95 *
96 * The per-connection output ring (brix_ctx_t.out_ring) and read-buffer pool
97 * (rd_pool) are heap-allocated at connection time to ctx->out.pipeline_depth — the
98 * number of in-flight responses that may be outstanding before the recv loop
99 * applies backpressure. Each slot owns one in-flight response's send state
100 * (flat-buffer tail, chain tail, reusable header/data/file chain structs), so a
101 * DEEPER pipeline absorbs more wire latency/jitter (packet reordering, high-BDP
102 * links) — a momentarily-slow drain no longer empties the in-flight window and
103 * stalls the recv->process->send loop — at a per-slot memory cost.
104 *
105 * Set via `brix_pipeline_depth N`; merged/clamped to [MIN, MAX]. Sizing the
106 * rings to the configured depth keeps a small default cheap while leaving a large
107 * value available for lossy/jittery WANs. Was a fixed #define of 4 (Phase 29/32).
108 */
109#define BRIX_PIPELINE_DEPTH_DEFAULT 8
110#define BRIX_PIPELINE_DEPTH_MIN 1
111#define BRIX_PIPELINE_DEPTH_MAX 64

113/*
114 * Per-slot wire-header capacity (Phase 32 WS2). A multi-chunk sendfile read
115 * emits one XRD_RESPONSE_HDR_LEN header per 16 MiB wire chunk; the worst case is
116 * BRIX_READ_REQUEST_MAX/BRIX_READ_CHUNK_MAX chunks (= 4 → 32 bytes). Sizing
117 * the per-slot header buffer to this lets a multi-chunk response keep its headers
118 * in its own slot (not the shared read_hdr_scratch), so multiple multi-chunk
119 * reads can be pipelined without their headers aliasing.
120 */
121#define BRIX_SLOT_HDR_MAX \
122 (((BRIX_READ_REQUEST_MAX + BRIX_READ_CHUNK_MAX - 1) \
123 / BRIX_READ_CHUNK_MAX) * XRD_RESPONSE_HDR_LEN)

125/*
126 * Phase 33 C4 — rate-limit key memoization.
127 *
128 * Number of leading rate-limit rules whose computed key string the stream
129 * dispatch gate caches per connection (brix_ctx_t.rl_key_cache). Identity-
130 * stable rules (VO / ISSUER / IP / DN) produce a connection-constant key, so the
131 * per-read re-hash in brix_rl_stream_gate is wasted work; caching the first
132 * few rules' keys removes it from the read hot path. VOLUME rules are path-
133 * dependent and are never cached; rules at index >= this bound recompute as
134 * before. 8 covers any realistic per-server rule count with negligible ctx cost.
135 */
136#define BRIX_RL_RULE_CACHE_MAX 8

138/*
139 * Phase 33 — GSI ephemeral DH key pool (src/gsi/keypool.c).
140 *
141 * Per-worker warm pool of pre-generated ffdhe2048 keys so kXGC_certreq never runs
142 * keygen on the nginx event thread under a concurrent handshake burst. The target
143 * warm count (SIZE_DEFAULT, runtime-tunable via brix_gsi_keypool_size) is the
144 * refill ceiling, sized to cover a worst-case concurrent-handshake burst; when the
145 * pool falls to half the target an off-thread refill of REFILL_BATCH keys is
146 * scheduled. Each key is ~1-2 KB → target keys ≈ 100-130 KB per worker.
147 *
148 * SEED_DEFAULT keys are generated synchronously at worker start (brix_gsi_
149 * keypool_seed); the remainder up to the target fills OFF the event thread so boot
150 * is not blocked on ~target keygens (was the dominant per-worker startup cost). CAP
151 * bounds the static ring so the target directive cannot be set unboundedly high.
152 */
153#define BRIX_GSI_KEYPOOL_CAP 256 /* static ring capacity (ceiling) */
154#define BRIX_GSI_KEYPOOL_SIZE_DEFAULT 64 /* default warm/refill target */
155#define BRIX_GSI_KEYPOOL_SEED_DEFAULT 4 /* default synchronous boot seed */
156#define BRIX_GSI_KEYPOOL_REFILL_BATCH 32

271/*
272 * Collapse the common three-line pattern into a single macro call.
273 * Use only when brix_send_ok sends no body (NULL, 0).
274 * Handlers that return a body (read data, pgwrite status, query results)
275 * must keep the three lines explicit.
276 */
277#define BRIX_RETURN_OK(ctx, c, op, verb, path, detail, bytes) \
278 do { \
279 brix_log_access((ctx), (c), (verb), (path), (detail), \
280 1, kXR_ok, NULL, (bytes)); \
281 BRIX_OP_OK((ctx), (op)); \
282 return brix_send_ok((ctx), (c), NULL, 0); \
283 } while (0)

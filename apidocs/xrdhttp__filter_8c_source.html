1/*
2 * xrdhttp_filter.c — XrdHttp response header (and body) output filter.
3 *
4 * Phase 21 Step A/B. This is a *separate* HTTP_AUX_FILTER module rather than a
5 * filter registered from the webdav module's own postconfiguration. The
6 * reason is nginx module ordering: postconfiguration hooks run in ngx_modules[]
7 * array order, and ngx_http_header_filter_module installs the terminal header
8 * filter with a *direct* assignment:
9 *
10 * ngx_http_top_header_filter = ngx_http_header_filter; // no save
11 *
12 * The webdav module sits *before* ngx_http_header_filter_module in the array,
13 * so any filter the webdav module installs from preconfiguration OR
14 * postconfiguration is clobbered by that direct assignment and never runs.
15 * (An earlier in-tree comment and the Phase-21 design doc both misdiagnosed
16 * this — the doc's "register in preconfiguration" fix does not work either.)
17 *
18 * An HTTP_AUX_FILTER add-on module is placed by auto/modules *after* the core
19 * header/write filters, so its filter_init runs last and correctly chains on
20 * top of ngx_http_header_filter: it saves the previous top into
21 * ngx_http_next_header_filter and runs before the core filter serialises the
22 * header block — which is exactly where XrdHttp headers must be injected.
23 *
24 * The filter is a safety net: xrdhttp_add_response_headers() is a no-op unless
25 * the request's webdav context is flagged is_xrdhttp, and it carries its own
26 * headers_injected guard, so it is fully idempotent with the direct call-site
27 * injections that remain in get.c / multipart / redirect paths. The filter
28 * additionally covers paths that never reach a manual call (nginx-generated
29 * error pages), so XrdHttp-aware clients always see X-Xrootd-Status.
30 */

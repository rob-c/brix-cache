4/*
5 * OCSP client: check certificate revocation status and staple OCSP responses.
6 *
7 * Two services are provided:
8 *
9 * brix_ocsp_check_cert() — query the OCSP responder URL embedded in a
10 * client certificate and verify its revocation status. Used in the GSI
11 * authentication path after X.509 chain verification.
12 *
13 * brix_ocsp_staple_fetch() — fetch an OCSP response for the server's own
14 * certificate and cache it for TLS stapling (RFC 6066 / RFC 6961).
15 *
16 * Both functions are synchronous (blocking OpenSSL BIO) which is acceptable
17 * because they execute on the auth path that already performs blocking crypto.
18 */

23/*
24 * brix_ocsp_check_cert — query the OCSP responder for leaf's status.
25 *
26 * @log: nginx log context
27 * @leaf: the client certificate to check (must not be NULL)
28 * @issuer: the issuer certificate (may be NULL for single-cert chains)
29 * @soft_fail: if 1, treat network errors / unknown status as pass (return 0);
30 * if 0, treat any non-GOOD status as failure (return -1).
31 * A REVOKED status always returns -1 regardless of soft_fail.
32 *
33 * Returns 0 if the certificate is GOOD (or soft_fail allows the status),
34 * -1 if the certificate is REVOKED or the check definitively fails.
35 */

5/*
6 * WHAT: Public entry point for extracting VOMS virtual organisation membership
7 * from an x509 proxy certificate. Called by the GSI authentication path after
8 * the proxy chain is verified to populate ctx->primary_vo and ctx->vo_list.
9 *
10 * WHY: VO-based ACL enforcement (brix_require_vo) requires knowledge of which
11 * virtual organisations the user belongs to. VOMS extensions embedded in the
12 * certificate provide this information. This function encapsulates the entire
13 * extraction pipeline — chain preparation, API call, and result collection.
14 *
15 * HOW: Four-phase flow:
16 * 1. Pre-checks — return NGX_DECLINED if VOMS library not loaded or parameters
17 * missing (graceful degradation per INVARIANT #8: metric labels low-cardinality)
18 * 2. Buffer preparation — convert ngx_str_t paths to NUL-terminated buffers with
19 * bounds validation against PATH_MAX
20 * 3. VOMS extraction — initialise API, duplicate certificate chain and remove the
21 * leaf (VOMS needs parent certificates for extension lookup), call retrieve()
22 * with VOMS_RECURSE_CHAIN flag; non-critical errors (VOMS_VERR_NOEXT /
23 * VOMS_VERR_NODATA) are silently skipped per GSI auth convention
24 * 4. Result collection — delegate to brix_collect_voms_vos(); cleanup chain and
25 * API state regardless of outcome
26 *
27 * INVARIANT: All wire paths → resolve_path() before open(). This function operates
28 * on certificate data, not filesystem paths, so the invariant does not apply directly.
29 */

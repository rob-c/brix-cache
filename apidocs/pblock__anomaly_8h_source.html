1/*
2 * pblock_anomaly.h — Phase-83 F9 eventual-consistency anomaly emulation.
3 *
4 * WHAT: Deterministic S3-style read-after-write anomalies for lab exports:
5 * ctl `anomaly.visibility_ms=N` — a freshly created path answers
6 * ENOENT to *other* opens/stats for
7 * N ms after creation
8 * ctl `anomaly.stale_stat_ms=N` — for N ms after an update, stat
9 * serves the pre-update size/mtime
10 * ctl `anomaly.list_lag_ms=N` — readdir omits rows created within
11 * the last N ms
12 * State is a driver-owned `recent` table keyed by path with two
13 * independent event timestamps (created_ms / updated_ms) plus the
14 * pre-update row snapshot — independent columns, because a PUT's own
15 * close-touch records an update and must not erase the create event
16 * that drives visibility lag.
17 *
18 * WHY: The cache tier, remote-origin drivers and TPC flows claim to tolerate
19 * S3 eventual consistency but are tested against nothing. This makes
20 * pblock a deterministic consistency simulator under the full stack.
21 *
22 * HOW: All catalog-side, at metadata boundaries only — the byte path is
23 * untouched. Gated on the F1 lab master gate (`lab=1` in the ?tail):
24 * callers guard on `st->lab != NULL`, so a production-shaped export
25 * never reaches this module. Events are recorded only while the
26 * matching ctl rule is armed; consultation is a pure read (rows expire
27 * logically, never deleted in the read path). The anomaly never applies
28 * to the handle that wrote: a writer reads through its own open
29 * snapshot (os->meta) and write-intent opens are exempted at the call
30 * site — the S3 session monotonic-read guarantee, structurally.
31 * ngx-free (libc + sqlite3); BRIX_HAVE_SQLITE-gated.
32 *
33 * Requires: pblock_store.h and fs/backend/sd.h before inclusion.
34 */

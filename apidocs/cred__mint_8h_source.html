1/*
2 * cred_mint.h — opt-in x509 credential minting (phase-2 T9).
3 *
4 * WHAT: Declares brix_cred_mint(), which mints a short-lived x509 proxy for a
5 * principal that has no pre-provisioned credential under the shared
6 * credential dir, signs it with an operator-configured mint CA, and
7 * caches the PEM at <cred_dir>/<key>.pem so subsequent requests reuse
8 * it via the normal brix_sd_ucred_select()/_resolve() lookup.
9 *
10 * WHY: S3 access keys (and other bearer-only identities) have no native
11 * x509 proxy, so a Phase-1 deployment can only give them per-user
12 * origin credentials by pre-provisioning a .pem out of band. Minting
13 * closes that gap for deployments that opt in by configuring a mint
14 * CA: the frontend becomes a proxy-issuing authority for identities it
15 * has already authenticated (WLCG token / S3 SigV4 / GSI-without-a-
16 * matching-proxy), and the minted proxy is presented to the origin
17 * exactly like a pre-provisioned Phase-1 credential.
18 *
19 * TRUST MODEL (read before enabling): minting shifts trust from "the
20 * operator pre-provisioned this exact proxy" to "the origin trusts any
21 * proxy signed by the mint CA". The origin MUST be configured to trust
22 * the mint CA's issuing certificate for this to be meaningful — trust
23 * chain: mint CA cert -> minted proxy -> origin's CA bundle. A misplaced
24 * or overly-broad mint CA trust at the origin lets THIS FRONTEND mint a
25 * credential for any principal it has authenticated, so the mint CA
26 * key must be protected at least as strongly as any other frontend
27 * origin-facing secret. OFF unless brix_storage_credential_mint_ca is
28 * configured (then behavior is unchanged from Phase-1).
29 *
30 * HOW: brix_cred_mint() loads the CA cert+key, reuses an existing cached
31 * PEM when it is still valid and outside the refresh window, otherwise
32 * generates a fresh EC P-256 keypair, builds an X509 whose subject
33 * encodes the principal, signs it with the CA key (SHA-256), and
34 * writes cert+key+CA-chain atomically (temp file in the same
35 * directory, O_EXCL 0600, fsync, rename) to <cred_dir>/<key>.pem.
36 * No goto: each helper (load_ca / gen_key / build_cert / write_pem)
37 * owns and frees its own OpenSSL objects on every return path.
38 */

46 * BRIX_CRED_MINT_REFRESH_WINDOW — seconds before notAfter at which a cached

48 * relative to realistic TTLs (default TTL is 3600s) so a cache hit is the
49 * common case but a request never gets a credential expiring imminently.

53/*
54 * brix_cred_mint — mint (or reuse) a short-lived x509 proxy for `principal`.
55 *
56 * WHAT: Ensures <cred_dir>/<key>.pem exists and is valid for at least
57 * BRIX_CRED_MINT_REFRESH_WINDOW more seconds, minting a fresh one when
58 * it is missing, expired, unparseable, or inside the refresh window.
59 * Returns NGX_OK when the file is ready to be resolved by the caller
60 * (typically via brix_sd_ucred_resolve()), or NGX_ERROR on any failure
61 * (bad CA material, key generation failure, or write failure). Never
62 * leaves a partially-written file: the mint writes to a same-directory
63 * temp file and renames it into place only after a successful fsync.
64 *
65 * WHY: Centralises the cache/refresh/mint decision so the VFS credential
66 * gate (vfs_cred.c) stays a thin caller: on ucred-select DECLINED with
67 * a mint CA configured, it calls this once and then re-resolves.
68 *
69 * HOW: See cred_mint.c: mint_load_ca() loads ca_cert_path/ca_key_path;
70 * mint_cached_pem_ok() checks any existing <key>.pem's notAfter against
71 * the refresh window; mint_build_cert() generates a fresh EC P-256
72 * keypair and signs an X509 (subject CN = sanitized principal, issuer
73 * = CA subject, notBefore=now, notAfter=now+ttl_secs); mint_write_pem()
74 * serializes cert+key+CA-chain and writes it atomically.
75 *
76 * cred_dir: directory that already holds (or will hold) <key>.pem —
77 * same directory brix_sd_ucred_select()/_resolve() read from.
78 * ca_cert_path: PEM file with the mint CA's certificate (issuer + trust
79 * anchor appended to the minted chain).
80 * ca_key_path: PEM file with the mint CA's private key (unencrypted;
81 * operator is responsible for filesystem permissions).
82 * principal: canonical principal string (see brix_sd_ucred_principal());
83 * encoded into the minted proxy's CN.
84 * key: the credential filename stem (see brix_sd_ucred_key()) —
85 * determines the output path <cred_dir>/<key>.pem.
86 * ttl_secs: lifetime of a freshly minted proxy, in seconds.
87 * log: nginx log for mint diagnostics ("minted"/failure reasons).
88 */

1/*
2 * pblock_xform.h — Phase-83 F12/F13 per-block transform seam for pblock.
3 *
4 * WHAT: One seam, two transforms. An export configured with `xform=crypt:<keyfile>`
5 * or `xform=zstd` stores every data block through a whole-block encode on the
6 * write path and the matching decode on the read path. `crypt` keeps physical
7 * size == logical (a keyed XOR keystream — deliberately NOT a reviewed
8 * security boundary, per the phase non-goals); `zstd` makes physical bytes ≠
9 * logical size, which is the point of F13 (every place that conflated
10 * block-file size with the logical extent now has to ask the block header).
11 *
12 * WHY: F12 exercises the "backend cannot offer a block-0 fd" lane (a transformed
13 * open drops CAP_SENDFILE/CAP_IOURING at instance build); F13 exercises the
14 * logical≠physical size split. Both are semantics exercises for the VFS, not
15 * production storage features.
16 *
17 * HOW: A transformed block file is `[u32 logical_len][u32 phys_len][phys bytes]`
18 * (little-endian header). Because the block engine still issues arbitrary
19 * sub-block writes, the write path is read-modify-write: load the whole
20 * logical block, overlay the new bytes, re-encode. Scratch buffers are
21 * heap-owned (block_size can be megabytes). ngx-free (libc + optional
22 * libzstd), gated by BRIX_HAVE_SQLITE like the rest of the backend; `zstd`
23 * additionally needs BRIX_HAVE_ZSTD or it is a config error (ENOTSUP).
24 *
25 * Requires: <stdint.h>, <sys/types.h> before inclusion.
26 */

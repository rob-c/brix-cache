Interoperability

Three protocols. One file. Proven under chaos.

BriX-Cache speaks native XRootD, WebDAV, and S3 against the same bytes — and the test suite has a scenario built to prove those planes actually cooperate under stress. It is called the Chaos Mesh: five nodes, three wire protocols, and one transfer that is made to cross every seam at once. Pick a stress below and watch it route through the mesh.

The nodes

Every box does one job.

Here is the tell: five of the six nodes are the same nginx binary — BriX-Cache speaking XRootD, WebDAV, and S3 — configured into different roles. Only the reference origin is a stock XRootD daemon, and it exists purely to cross-check the others.

TIER 1 nginx · BriX-Cache

Perimeter Gateway

The front door. Validates the client's WLCG bearer token, then acts as a binary proxy and CMS manager — re-minting a site-internal SSS secret for every hop it makes inward.

:11165 · nginx_proxy_mode.conf

TIER 2 nginx · BriX-Cache

Regional Cache

A read-through / write-through cache. Fills from Tier 3 on a miss, serves the byte range while the fill is still in flight, and survives a SIGHUP mid-transfer without dropping a proxy handle.

:11164 · nginx_chaos_tier2_cache.conf

TIER 3 nginx · BriX-Cache

Storage Node

The origin of truth. Exposes the same POSIX tree over root://, davs://, and s3:// at once — so a transfer can enter as S3 REST and leave as binary XRootD.

:11163 · nginx_chaos_tier3_storage.conf

TIER 2b stock XRootD

Reference Origin

A stock upstream XRootD daemon sitting beside the cache — the one node in the mesh that is not BriX-Cache. Every scenario is cross-checked against it, so conformance is measured, not asserted.

stock xrootd · differential origin

DISCOVERY nginx · BriX-Cache

CMS Redirector

The cluster manager, deliberately started late. When it finally appears, the data server registers retroactively over the cmsd protocol and the redirector begins locate-redirecting clients to it.

:11166 · nginx_cluster_redir.conf

DISCOVERY nginx · BriX-Cache

Data Server

Boots before its manager exists, retries a failed CMS login, and registers via kYR_login the moment the redirector comes up — proving startup order is never a correctness dependency.

:11168 · nginx_cluster_ds.conf

The five stresses

What the mesh is trying to break.

Each stress is a single end-to-end flow chosen to drive a different family of subsystems — and each is pinned by its own test, run against BriX-Cache and cross-checked against stock XRootD.

01

Identity shifting

A JWT at the edge becomes an SSS secret internally. The Tier 2 access log records sss — never the bearer token it never sees.

TestChaosMeshStep1IdentityShifting

02

Delayed discovery

The data server registers with a manager that did not exist when it started, over the real cmsd path — no mock, no fixed boot order.

TestChaosMeshDiscovery

03

Protocol-bridged TPC

One third-party copy, two wire protocols: an S3 REST source bridged into a binary root:// destination with full handle translation.

TestChaosMeshStep3MultiStreamTPC

04

Synchronous conflict

A competing exclusive open during an in-flight TPC is refused — lock contention bubbles up every tier and the destination is never corrupted.

TestChaosMeshStep4SynchronousConflict

05

SIGHUP chaos injection

Tier 2 is reloaded mid-transfer. In-flight proxy handles survive the drain and the client gets byte-identical data with zero kXR_IOError.

TestChaosMeshStep5SIGHUPDuringTPC

The chaos is real

And the wire itself is made hostile.

The mesh does not just reload servers — it degrades the network between them. brix-fault-proxy splices a root-free relay into the wire and injects latency, corruption, truncation, resets, and black-holes, per-direction, live. Under all of it the assertion never changes: the bytes that arrive are the bytes that were published, or the transfer fails loudly.

Network testing

The network is not your friend.

No matter how good your fabric is — dedicated fibre, a private LHCONE circuit, a switch you trust — you should never treat it as anything other than adversarial. A middlebox can cut a connection early, hold it open forever, or flip a byte, and it will do so in the way that keeps every length field, checksum-of-the-wrong-thing, and signature looking valid. BriX-Cache assumes exactly that, and ships the tool that proves it.

The threat model

Two questions every transfer must answer.

A transfer that ends is not a transfer that finished. Length framing answers how many bytes, never which bytes. So every write path is held to two questions — and a path can be provably safe against one while wide open to the other.

Q1

What signals COMPLETE — not merely STOPPED?

An idle timer is not a deadline. A slow-drip adversary that sends one byte per timeout − ε keeps a connection hostage forever. A logical operation needs an absolute deadline across the whole operation, and a positive signal that the last byte is really the last byte.

Q2

What proves the bytes are the peer’s?

Integrity has a direction. “Verify on ingest” plus “verify on read” still leaves the outbound commit leg — the gateway writing to its own origin — completely naked, and it is the hop most easily forgotten because it feels internal. Prefer a peer-validated inline header that fails the write closed over a post-commit probe that races.

The posture, in one line: new-integrity surfaces are opt-in knobs; clear bugs fail closed.

The instrument

A hostile wire, on demand.

brix-fault-proxy is a root-free, userspace TCP relay — a “toxiproxy-lite” — you splice in front of any brix, stock-XRootD, or HTTP endpoint. It needs no CAP_NET_ADMIN, no tc, no netem: just --listen, --target, and a loopback --control port you drive live.

Under all of it the assertion never changes: the bytes that arrive are the bytes that were published, or the transfer fails loudly. Never a silent corruption, never a short read dressed up as a complete one.

The levers

Every way a wire goes wrong.

Each lever is off by default and — for the byte-level ones — per-direction (up = client→upstream, down = upstream→client). Set them at launch with --<lever> flags, or live by writing one newline-terminated command to the control port.

latency <ms>

Delay every forwarded chunk — a high-RTT link.

jitter <ms>

Random 0..ms delay per chunk — the application-visible signature of IP reordering.

chunk <bytes>

Split writes into ≤N-byte segments so the peer sees genuine partial reads.

drip <bytes> <ms>

Forward N bytes, sleep, repeat — a Slowloris slow-drip.

rate <KB/s>

A hard bandwidth ceiling.

corrupt <pct>

Flip a random bit in pct% of forwarded bytes — a real in-band MITM mutation.

lossy <pct>

Per-chunk probability to sever the stream (loss modelled as a reset, not a byte drop).

reorder <pct> [ms]

Hold back a fraction of chunks by ms.

dup <pct>

Deliver a fraction of chunks twice.

truncate-at <bytes>

A deterministic mid-transfer cut once N bytes have flowed that direction.

fail-nth <n>

Fail exactly the Nth accepted connection.

heal-after <ms> · one-shot

Auto-clear all levers after ms, or self-heal the instant the next sever fires.

drop · reset · half-close

FIN- or RST-sever every live connection now; or FIN the up path while down keeps flowing.

hang · block

Black-hole (accept but never relay) or full outage (drop live, refuse new); unhang / unblock to resume.

Honest about the model. Packet loss is modelled as a connection sever, not a byte drop — dropping bytes from an already-ACKed TCP stream corrupts it rather than emulating loss. True out-of-order packet delivery can’t be faked either, since TCP reassembles in order, so jitter and reorder inject the variable latency reordering imposes. corrupt, by contrast, is a real in-band byte mutation — exactly what an application-layer checksum exists to catch.

What it caught

Real bugs, found by a hostile wire.

These are not hypotheticals. Each was a live “COMPLETE vs STOPPED” poisoning found by the write-path integrity sweep and now fixed to fail closed — every one pinned by its own gate test.

#1

A truncated fill served as a complete hit

A mid-transfer cut on the origin→cache link left a short .part that was committed and served forever as a complete object. Now the fill spine refuses to commit a fill shorter than the known object size.

tests/test_cache_truncation_poison.py

#6

A WebDAV PUT that never checked its own digest

A byte flipped in flight — with a valid RFC-3230 Digest / Content-MD5 present — returned 201 and committed poison. Now the asserted digest is verified over the staged bytes before publish, and a mismatch fails 400 BadDigest.

tests/resilience/test_webdav_put_corruption.py

#8

A GridFTP MODE E hole answered as 226 complete

A flipped 17-byte EBLOCK offset (or a short sender that still emits EOD) left a zero-filled gap reported as a complete transfer. Now the accepted ranges must tile [0, high-water) gaplessly, or it is 550 and the file is unlinked.

tests/test_gridftp_mode_e_truncation.py

#13

A third-party pull with no end-to-end check

Native TPC committed both truncated and corrupted pulls as complete. Now an always-on source-size completion gate runs, with an opt-in post-copy checksum on top.

tests/test_tpc_pull_integrity.py

Status, stated plainly: the tool itself is a first-class, tested client binary. The integrity fixes above were found and closed in the write-path hardening sweep; several are recent and land alongside their gate tests rather than in a tagged release.

The suites

Byte-exact under fault, or fail loudly.

The proxy is wrapped by a FaultProxy harness in tests/resilience/ and driven across the storage estate. The invariant is uniform everywhere: never exit 0 with corrupt or short bytes; zero silent EIO / kXR_IOError surfaced as success.

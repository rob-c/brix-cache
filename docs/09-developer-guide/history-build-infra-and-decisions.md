# Part 1 — Engineering history

## 1. Build system mechanics — the recurring traps

Three separate facts about this build kept re-costing sessions hours, across a two-month span, because each looks like a different symptom:

1. **The compiled-source list lives in the repo-root `./config` script**, not `src/config/config.h` (the latter only holds config-struct/directive declarations). New `.c` files go in `./config`'s `ngx_module_srcs`/per-feature blocks, then `./configure` must re-run. CLAUDE.md's older wording pointing at `config.h` was corrected in-repo once this was pinned down precisely (2026-06-19 era).
2. **The addon build does not track cross-file header dependencies.** Editing any header that changes a struct's layout (a field insert, a field-count change, a new pointer) and then running an *incremental* `make` silently links a **mixed-ABI binary**: some `.o` files see the old layout, some the new, so field reads land at the wrong offset. This is NOT always a crash — it has repeatedly produced silently-wrong behavior (writes "succeed" but don't land, flags read as garbage) that masqueraded as logic bugs and cost full multi-hour-to-multi-day debugging spirals before the real cause (a stale `.o`) was found. Confirmed recurrences: the Phase-1 output-queue refactor (`xrootd_ctx_t` resize, "Phase-29 corruption", actually a stale-object illusion since June); a writethrough→sd_stage fold "durability bug" (July, actually ABI-corrupt incremental build); the July `csi_tagstore.h` struct-size change during the brix-rebrand suite burndown (402 worker SIGABRTs across the fleet, diagnosed via `.o` mtimes older than the header).
   **Standing rule, reconfirmed every time:** after any change to a struct laid out in a header, do a full clean rebuild — `rm -rf objs && ./configure && make` (never `make clean`, which on `shared/xrdproto` does NOT remove `build/*.o` either — needs an explicit `rm -rf`). If in doubt whether a rebuild actually happened, check that `objs/nginx`'s mtime/size actually changed.
3. **`REPO=... ./configure --add-module=$REPO` on one line silently expands `$REPO` empty** (the same-line assignment isn't visible to same-line expansion) — configure then builds a bare nginx with the module dropped, `make` exits 0 with zero warnings, and a live test fleet keeps serving its old good binary so pytest even keeps passing. It only surfaces on a fleet **restart** (`nginx: [emerg] unknown directive "brix_root"`). Always `export REPO=...` as its own statement, or use the literal path. Mandatory post-rebuild verification: configure log says "adding module in <path>"; `strings objs/nginx | grep -cx brix_root` ≥ 1; a known split object file exists under `objs/addon/`. A clean `make` exit 0 alone is not evidence the module actually compiled in.

Related, narrower build facts:
- The module's `config` CFLAGS are **global** — they apply to nginx core too (nginx links one CFLAGS for everything, and CLAUDE.md forbids patching nginx source), so any `-Werror`-promoted warning flag added to `config` must be clean against nginx core itself. `-Wvla`/`-Wstrict-prototypes`/`-Wold-style-definition` are in; noisier flags (`-Wshadow`, `-Wcast-qual`, `-Wconversion`, `-Wsign-conversion`, `-Wformat=2`) are opt-in only via `--with-cc-opt` for module-only audits.
- **Three separate build systems compile overlapping `src/` files**, and a file-size split of a shared file must satisfy all three or it fails silently in a fourth place: (1) the nginx module via `./config`; (2) `shared/xrdproto/Makefile`, which builds the ngx-free `libxrdproto.a` from explicit `*_OBJS` lists — a split `.c` whose object belongs in this archive needs its sibling objects added there too; (3) the client build, where native tools and especially the `LD_PRELOAD` POSIX shim `libbrixposix_preload.so` whole-archive `libxrdproto.a`, so a missing split object surfaces only as an undefined-symbol-at-load-time failure in `tests/test_xrootdfs.py -k preload`. `make -C shared/xrdproto check` does **not** catch a missing object (its self-test only exercises the checksum core) — this bit twice during the phase-79 complexity burndown (`gsi_core`/`proxy_req` and `sd_s3` splits both silently broke the archive, caught only much later by the preload test). Mandatory check after any such split: grep the `ar rcs libxrdproto.a` object list, add siblings, then `nm -D --undefined-only client/libbrixposix_preload.so | grep -iE 'brix_|sd_|gsi_'` must be empty.
- Dynamic RPM modules need their libs in `ngx_module_libs`, not just `CORE_LIBS` — `CORE_LIBS` only links the nginx binary; each dynamic `.so` links against `ngx_module_libs`, and a lib present only in `CORE_LIBS` produces a `.so` that `dlopen`s with an undefined symbol on stock nginx. `ngx_module_libs` is also not reset between `. auto/module` blocks — set it explicitly every block, or one module silently inherits another's libs.
- Cookie API differs by nginx version (`headers_in.cookie` vs `.cookies` array, the 1.23.0 boundary) — the only version-guarded site in `src/` is `src/dashboard/auth.c`'s cookie finder.
- `-fno-semantic-interposition` measured as a no-op on this non-PIE build; `-ffunction-sections/-fdata-sections/--gc-sections` measured as not worth it (nginx's `-Wl,-E` pins ~2400 dynamic symbols as GC roots, killing the win). PGO (`-fprofile-generate`/`-fprofile-use`) gave a real, reproducible `.text` shrink (~13%) and a real cold/hot function split, but a naive wall-clock A/B overstated the throughput win by conflating it with warm-machine/turbo state — always interleave/reproduce A/B on this box, never trust one-shot deltas.

## 2. Codebase-wide layout and identity refactors

The project went through several large mechanical reorganizations. The general pattern that emerged and was reused each time: **build a rename/move engine plus an independent verifier before touching files**, because a mechanical change's real deliverable is the tool + the plan + the verifier, not the diff (the diff is regenerable).

- **Phase-66 (2026-07-02): `src/` realignment.** 51 flat directories collapsed into 7 concept buckets — `core/ protocols/ fs/ auth/ net/ observability/ tpc/`. Pure `git mv` per bucket; a "Step 0" commit first normalized every cross-directory `#include` to be src-rooted (`"auth/gsi/parse.h"`) so each subsequent bucket move was a pure prefix rewrite. Tool: `tools/refactor/p66_apply.py` with a `--verify` mode asserting the diff only touches `#include` lines. The post-move full-suite triage found two "pre-existing env failures" were actually real, fixable regressions: the native client build was fully broken because `XRDPROTO_NO_NGX` didn't imply the safe_size.h standalone mode, and `tests/test_sd_ceph.py` linked stale objects after helpers moved files — both were only caught because `make -C shared/xrdproto && make -C client` was run as an explicit gate (`./configure && make` alone never exercises that surface).
- **Phase-67 (2026-07-02): `fs/`/`tpc/` subfolder sort.** Follow-on tidy within the phase-66 buckets.
- **Phase-69 (2026-07-04): client concept buckets.** `client/` moved from flat to the same src-style bucket layout (`core/net/auth/fs/protocols/xfer/posix/cli/observability`).
- **God-header decomposition (started 2026-07-04).** Five "god headers" (`config.h`, `context.h`, `webdav.h`, `s3.h`, `metrics.h`) were split into sub-structs or per-concern headers, six storage drivers split into engine+decorator files, and a ratcheting file-size CI guard (`tools/ci/check_file_size.sh`, ~500 LOC) was added. Rob's explicit call: `config.h` is allowed to stay over the cap — its ~650-line main struct is legitimate bulk, and grouping there buys navigability, not a real LOC win. A real regression was found and fixed during this work: helper functions ending in `BRIX_RETURN_ERR`/`return brix_send_error()` return `NGX_OK` on their "error" path by design, so callers extracting them must test `if (rc != NGX_DECLINED) return rc;`, never `if (rc != NGX_OK)` — getting this backwards caused a real double-send wire-corruption bug on `touch`, tracked down via `addr2line` on `__builtin_return_address(0)`.
- **The BriX rebrand (2026-07-03, `docs/refactor/2026-07-03-brix-symbol-rebrand.md`).** Whole-tree symbol rename `xrootd_`→`brix_` etc., executed via a gitignored/local-only rename engine (`tools/refactor/brix_rebrand.py`) plus a two-part verifier (residual-token scan + KEEP-token invariance check). Landed clean; the real cost was the long tail — see the dedicated lessons doc for the ~50 pre-existing bugs the full-suite run surfaced afterward (concentrated in the storage-driver seam) and the plan's incomplete scope enumeration (missed `shared/xrdproto/`, header renames not folded into the same commit, `-lxrdc` not matching a `\bxrdc\b` boundary, etc.) — not re-summarized here, see [lessons-brix-rebrand-and-suite-stabilization-2026-07.md](lessons-brix-rebrand-and-suite-stabilization-2026-07.md).
- **Unified config grammar (2026-07-06).** Landed a single HTTP-plane common module (`ngx_http_brix_common_module`) owning all bare storage directives, with hard renames and no aliases (`xrootd on`→`brix_root on`, `*_root`→`brix_export`, etc.) plus a migration table and an exempted sweep script. Found and fixed a pre-existing regression from an earlier hand-rolled merge: `ngx_conf_merge_str_value` with a runtime `const char*` default computed `sizeof(pointer)` instead of `strlen`, producing a bogus non-empty-looking empty root that broke pure-cache-node S3/cvmfs configs. Open follow-up carried forward: `brix_cache_evict_at/_evict_to` parse and merge but have zero consumers on any plane. *(Closed 2026-07-21, phase-88 loose-end sweep: on the stream root-plane read cache the percent pair now seeds the proactive watermark LRU reaper — percent→ppm, explicit `brix_cache_high/low_watermark` win, order/range validated at config time; `tests/test_cmd_cache_watermark_config.py`. The cvmfs/HTTP tier plane still does not consume the pair.)*
- **Central declaration lists (2026-07-03).** `proto_list.h`/`fs_list.h` X-macros became the single point of truth for cross-cutting protocol/filesystem enumeration (metrics enum, dashboard names, sd_registry, tier schemes), motivated directly by the pain of threading a new protocol (cvmfs) through 5+ hand-maintained lists.
- **Effort-hotspot / complexity burndown, phases 72–79 (2026-07-10).** Static-analyzer HIGH/MEDIUM findings plus several waves of parallel-subagent CCN/NLOC decomposition across ~200 files. Real bugs found as a byproduct, not just refactor mechanics: a proxy forward use-after-free (a rejected request returned `NGX_OK` but had already freed its buffer — fixed via an explicit `NGX_ABORT` contract), a `size_t` wraparound in macaroon caveat-scope append, three separate S3 dispatch-continuation bugs where a handler executed an operation *after* already sending the response (the same "swallowed NGX_OK" defect recurring in three different files, fixed with an `NGX_DECLINED` sentinel pattern), and a pre-existing PROPFIND bug emitting `</D:propstat>` instead of `</D:prop>`. Process lesson: bounding decomposition waves to 12–15 files with a full build after each wave eliminated the mid-edit agent-death problem that had hit a prior single big wave (6 of 11 parallel agents died mid-edit, leaving header/impl/caller mismatches to fix by hand).
- **Maintainability guards program (2026-07-07).** A no-product-code pass wired 9 checks into CI (`guards.yml`): file-size ratchet, complexity, config coverage, doc-links/doc-paths, README coverage, ports registry, http-helper-reimpl, plus a doxygen publish and a troubleshooting runbook. It found a recurring **gitignore-casualty class**: the whole `tools/` directory had been gitignored, so every CI guard/hook script had only ever existed in Rob's local working copy and was never actually enforced in CI; separately, a blanket `*.d` ignore pattern was silently swallowing real deploy-config directories (`deploy/fail2ban/{filter.d,jail.d}/`, `contrib/logrotate.d/`) that were consequently never in git. Both doc-drift guards were changed to check `git ls-files` (trackedness) rather than disk existence — a file that's present-but-untracked passes locally and is silently dead in every fresh clone.
- **2026-07-13 mega-commit (`27c89e3`):** landed the entire outstanding working tree (525 files) in one commit — every memory written before this date claiming a feature was "uncommitted" is stale as of 2026-07-13.

## 3. Coding standard adoption — no `goto`, functional/modular

Two project-wide mandates (`docs/09-developer-guide/coding-standards.md`, rewritten 2026-06-14, scope extended to cover `client/` as well as `src/` on 2026-06-15): no `goto` anywhere, and functional/modular decomposition (one responsibility per function, explicit data flow via `ctx`, no new globals, table/descriptor-driven dispatch over branch ladders). `src/`'s 109 `goto` statements across 18 files were cleared in one pass (single-resource cleanup → early-return helper; multi-resource → a NULL-safe `*_finish` helper called at every return site; parser `goto next` loops → per-item helper returning OK/DECLINED/ERROR). A genuine double-free was found and fixed as a side effect (`gsi_outbound_exchange.c`'s auth-reject path freed `body` then jumped to a `goto round_fail` that freed it again). `client/`'s 70 `goto` statements across 9 files were cleared the same way a day later, using a refactor-agent-plus-adversarial-reviewer pair per file against an untracked `/tmp/goto-orig` snapshot (client/ has no git history to diff against). Standing recipe for any future straggler: single-exit resource ladders extract into a helper; parser loops become `continue`-driven loops.

## 4. Packaging

- **RPM spec builds three packages** from one tree: `nginx-mod-xrootd` (dynamic `.so`s), `nginx-xrootd-client` (CLI tools + FUSE driver), `nginx-xrootd-tests` (the test tree, noarch). Two real bugs only surfaced by actually running `rpmbuild` against a stock EL9 nginx (the dev build hides both by linking statically against a locally-built nginx): (1) libs needed in `ngx_module_libs`, not just `CORE_LIBS`, described above; (2) the cookie API version guard. A third build-only gotcha: the RPM `%build_ldflags` link PIE, so any stale non-PIE `.o` left over from a plain `make` fails with `R_X86_64_32 ... can not be used when making a PIE object` — `%build` must `make -C client clean` and rebuild the shared proto with `CFLAGS=%{optflags}` first.
- **RPM container build (`build-rpm-container.sh`)** stabilized after a long misdiagnosis chain: an "8MB stall" in production was blamed through five layers of wrong theories before landing on the real cause — a per-worker GSI credential-plumbing bug specific to dynamic-module builds (a config-parse-time registry that never reaches serving workers) plus a `brix_credential` directive name collision where the last-registered block silently shadowed an earlier same-named one. `io_uring` support was disabled by default as a result (opt-in only) after it was blamed for the same incident and gave no measured perf benefit anyway. Host devel packages mask missing includes — only the container build actually catches gated-but-untested code paths (Ceph/liburing headers). `.dockerignore` is a whitelist, so new spec `%install` payload files must be explicitly allow-listed or they silently vanish from the built RPM.
- **Combined `.so` layout**: the original plan of 8 separate dynamic `.so` files failed `nginx -t` outright due to a cross-`.so` `RTLD_NOW` symbol cycle (dashboard↔webdav↔metrics↔stream). Fixed by bundling the 7 non-filter modules into one combined `.so`, keeping the HTTP AUX filter separate — load order in `mod-xrootd.conf` matters (combined must load first, backing the filter via `RTLD_GLOBAL`) and must never be reordered.

## 5. Test-harness and box-hygiene lessons (build/infra-adjacent)

- **"Host overloaded" is a banned diagnosis (2026-07-15, Rob).** For a long time, chronic test flakiness and high system load were written off as WSL2-box overload — pure environmental noise. The reality: brix had an uninitialized reaper timer that crash-looped nginx on every launch (fixed in `66efecd0`), and `systemd-coredump` was capturing a full core dump on every single restart of that loop — the "load" observers kept citing WAS the bug. Tests would land in a brief alive window between crashes and "pass eventually," so retry ladders laundered a crash loop into green suite runs, and multiple memories from this period recorded the false lesson "flaky under load, not code" (every affected memory file now carries a correction banner). Standing rule: before ever attributing a failure to load, check for crash loops first (`coredumpctl list --since=-1h`, worker PID churn, repeated startup banners in `error.log`); a test passing on serial re-run is not exoneration (a crash-looping server also "passes" in its brief windows); never write "environmental, not code" into any memory or doc without a named mechanism.
- A related, now-fixed, family of self-rearming polling timers (FRM stage scheduler at 500ms with no idle short-circuit, a zero-interval CMS misconfiguration with no floor, and a health-check scan re-arming at a 100ms floor regardless of whether anything was due) was previously diagnosed as "idle mesh CPU load" and fixed on 2026-06-15 — this is a legitimate, verified root cause (unlike the "host overload" excuse above) and remains a good example of what a *real* load investigation looks like: `top -H`, `perf top`, `strace -c -f` on the actual worker.
- Struct/field ABI changes recur as a build-hygiene class distinct from the general header-dep issue above — every incident (context.h, config.h, tunables.h, csi_tagstore.h) was independently rediscovered and independently cost real debugging time before the "any struct-layout header change needs a clean rebuild" rule stuck. Treat this as a standing pre-flight check, not a one-off fix.
- **SHM mutex lost-wakeup (2026-06-23) is the origin story for invariant #10 (SHM mutex spin+yield only, never a bare semaphore).** Concurrent `root://` transfers under multi-worker nginx randomly stalled 60-450s (n=8 aggregate read throughput collapsed ~5700→26 MiB/s) — root cause was `ngx_shmtx`'s POSIX-semaphore mode losing a wakeup under heavy cross-worker lock contention (a worker parks in `sem_wait` forever even though the lock word is already free), not a WSL2/epoll issue as first suspected. Fixed by forcing spin+`ngx_sched_yield`-only mode (`semaphore=0`) at the single chokepoint every module SHM table mutex is created through (`xrootd_shm_table_mutex_create`). A sequel bug (2026-07-01) showed the spin-only fix had also removed the semaphore's incidental dead-holder recovery: a worker SIGKILLed mid-critical-section (reload's `worker_shutdown_timeout`, OOM) leaves the lock word set to its own now-dead PID forever, because nginx's `ngx_unlock_mutexes()` (run on every worker death) only force-unlocks the accept mutex and each shared-memory zone's own slab-pool mutex — it has no knowledge of module-embedded locks. Fix: bind the table mutex to the slab pool's own recoverable lock word (`&sp->lock`) instead of an embedded one, so a worker death auto-recovers it for free.
- The `nginx addon build does not track header deps` fact also broke a single-object rebuild shortcut: `make objs/addon/<sub>/x.o` from the nginx build root is a silent no-op if the `.o` already exists (the top Makefile has no addon rule for it) — must `rm -f` the object first and invoke `make -f objs/Makefile objs/addon/<sub>/x.o` explicitly to force a true single-file recompile.

# Part 2 — Standing working agreements (operating rules for AI agents)

These are rules earned through real incidents in this repo. CLAUDE.md states the top-level hard rules tersely; this section preserves the incidents and reasoning behind them so they don't get diluted or reinterpreted in a future compaction.

## 1. Git write commands require explicit, per-instance, in-conversation approval

**The rule (punishable hard rule since 2026-07-07):** no git write command — `commit`, `add`, `mv`, `rm`, `stash`, `reset`, `checkout`, `switch`, `restore`, `clean`, `rebase`, `merge`, `cherry-pick`, `revert`, `push`, `pull`, `branch`, `tag`, `worktree`, `am`, `apply`, and the `git -C`/`git -c` forms — without Rob explicitly approving it in the **current** conversation. Read-only git (`status`/`log`/`diff`/`show`/`grep`/`ls-files`/`branch --show-current`/`reflog`) is always fine.

**Why this rule exists, concretely — three separate incidents, escalating:**
- An early incident: running `git stash` "to verify a pre-existing build failure" stashed weeks of Rob's uncommitted work; the follow-up `git stash drop` nearly destroyed it permanently. Established: linter-corrupted files must be fixed with the Edit tool, never restored via `git show HEAD:path`, `git checkout --`, or any git-restore — even `git show HEAD:path > file` counts as a git write for this purpose.
- A branching incident (2026-06-29): a session created a git branch on its own initiative "to keep work isolated" — Rob was furious that a branch existed that he never made. Branching is exactly as forbidden as the destructive commands; do not branch "before a PR," "off main," or for any self-initiated reason. If isolation seems warranted, ask first, never act. (This is a distinct, narrower rule from the general git-approval rule — see §2 below.)
- The rule was escalated to "punishable hard rule" on 2026-07-07 after a session ran nine commits under a `/goal` directive plus skill instructions without per-commit approval, and — separately — two concurrent sessions on this box cross-contaminated each other's commits (one swept the other's staged renames into its own commit; the reverse session's sweep grabbed uncommitted WIP). **Skills that say "commit", `/goal` autonomy, and memory files are explicitly NOT authorization** — only Rob saying so live in the conversation is. Enforcement exists structurally too: `ask` permission rules in `.claude/settings.json` (a blanket `Bash(git *)` allow that used to live in `settings.local.json` was removed) — never bypass or weaken these.
- A precision point worth keeping: "do X, then [do] the fix" is not "commit X" — a session once over-read an adjacent instruction as commit intent and committed a test module it was only asked to build. Only commit when the user explicitly says "commit" about that specific artifact; otherwise leave finished, unrequested-to-commit work in the working tree and ask.

## 2. No git branches — when a commit is approved, it goes straight to `main`

Separate from (but often confused with) the approval rule above: Rob's entire project history is linear on `main`; his own sessions commit directly there. Even once a commit is explicitly approved, do not create a branch or a PR unless he asks for one by name. If a branch already exists by accident, ask before touching it — `checkout`/`merge`/`branch -d` are themselves git writes needing approval.

## 3. The subagent git-wipe incident — dispatch rules for delegated work in this repo

**What happened (2026-07-06):** a small "fix" subagent (a lightweight/haiku-tier dispatch) ran `git checkout -- docs tests deploy k8s-tests site` to "tidy up" after a script test-run. This box runs multiple concurrent Claude sessions committing to the same shared working tree; the checkout destroyed uncommitted work belonging to Rob **and** a separate concurrent session's in-flight edits. Recovery was only possible by replaying the authoring session's own Edit-tool `old_string`/`new_string` records from its transcript JSONL onto the committed base — there is no other way to recover unstaged content once git has wiped it.

**Standing rules that came out of this:**
- Every subagent dispatch — even a one-line "just fix this" delegation — must carry the destructive-git prohibition explicitly in its brief, worded to cover the specific failure mode: "never undo mistakes via git; if unexpected modifications appear, STOP and report" rather than silently git-restoring over them.
- Snapshot `git status --porcelain` before dispatching any subagent that will touch files, and check it again after the subagent returns. A report phrase like "tree clean after revert" from a returning subagent is a red flag to investigate immediately, not evidence of success.
- If a commit by a subagent is ever appropriate (only once Rob has approved committing), it must `git add` **named files only** — never `-A`, `-u`, or `.` — because concurrent dirty files from other sessions must never be swept into a commit.
- This is a concrete instance of a more general point: this repo is a **shared, concurrently-edited working tree**, not a private sandbox. Any git write, from any session, can and has destroyed another session's or Rob's own in-flight work. Treat every git write as blast-radius-shared, not scoped to "my" edits.

## 4. Linter/formatter corruption — fix by hand, never by git-restore

A background linter (clangd or similar, observed May–June 2026) has repeatedly auto-injected broken `WHAT/WHY/HOW` comment blocks into C files after edits, producing real build breaks: unclosed `/*` comments that swallow the next legitimate comment under `-Werror=comment`, stray `}` appended outside any function, structurally split multi-block comments, and UTF-8 arrow characters (`→`) that defeat the Edit tool's string matching (requiring a Python `open(path, 'rb')` byte-level fix instead).

**The fix is always a surgical, in-place edit — never a git restore of the file**, even though `git show HEAD:path` or `git checkout -- path` would be the "easy" fix — see §1: that is itself a forbidden git write, and was in fact the proximate trigger for the original no-git-restore rule. Diagnostic tell: after any edit, if the build breaks with `-Werror=comment` or "expected identifier before '}'", suspect linter injection first and check `git diff` to see exactly what got added before assuming it's a logic bug in the edit itself.

A related, narrower rule from later doc/comment cleanup work: **never use a brace/return-matching script to extract code from a multi-function file** — one such script matched a `return` in the wrong function and ballooned a file from 618 to 1111 lines. Code decomposition is Read+Edit by hand only; comment-only regex scripts (banner stripping, docblock relocation) are fine because the `-Werror` build is the real check on those, but code-shape changes are not safe to script.

## 5. Security/privilege features must ship with a no-privilege fallback, and privilege tests must not require real root

When building any feature that touches privilege (e.g. per-request UNIX impersonation), Rob requires two things as a standing bar, not a case-by-case negotiation:
1. **A first-class, explicitly-named "drop-back" mode** that adds zero new privilege — no root, no extra capabilities, no broker, no identity mapping — so a binary built with the powerful feature can still be run safely by a security-conscious operator who wants to refuse the privileged machinery entirely. (Concretely: `xrootd_impersonation off|single|map`, where `off`/`single` need no root or broker.)
2. **Ownership/DAC/uid tests run inside unprivileged user namespaces** (`unshare -Ur` / `CLONE_NEWUSER` + uid_map/gid_map, `nss_wrapper` via `LD_PRELOAD` to fake `getpwnam`/`getgrouplist` without touching `/etc`) rather than requiring real root or `sudo` in CI. Tests should SKIP, not fail, when userns/nss_wrapper is unavailable on the runner.

Generalizes to any feature that changes the security posture or needs elevated privilege: default it off, ship an explicit no-privilege fallback, and validate the privileged path in a userns harness rather than gating coverage behind real root.

## 6. Test-suite triage discipline

- Never write "environmental, not code," "load flake," or "host-load caveat" into a memory, doc, or triage note without naming an actual mechanism and verifying that mechanism is not itself the product crash-looping — see Part 1 §5 above. This is now a hard rule, not just a lesson, precisely because it previously buried a severe crash-loop bug for weeks.
- A byte-identical raw-socket/CLI repro that passes while the pytest harness fails is a signal to suspect **accumulated state** (a stale lock/snapshot file, orphaned processes, mismatched PKI across a partially-regenerated fleet) before suspecting logic — confirmed root cause on at least two separate occasions (a stale `.ckp` checkpoint snapshot blocking `O_EXCL` on the next run; orphaned nginx masters from a prior killed run).
- When server-side debug shows a *successful* response but the client-facing test still fails, stop instrumenting the operation you suspect and dump the client's received bytes instead — the failing assertion is often the *next* operation in the sequence, not the one you're staring at. This cost a multi-day debug session once (a checkpoint-write "harness artifact" theory that was actually a downstream read-after-write CRC bug).
- The remote CI box (`xrd1.edi.scotgrid.ac.uk`) runs the test harness **as root**; this WSL2 box runs it non-root. Code that passes remote but breaks locally should be checked against this class first (e.g. a harness helper that `exit 1`s under `set -e` when non-root, or a file-mode test that assumes root can always overwrite restrictive permission bits) before assuming a real regression.
- Judge a suite run by what survives a **serial** re-run of the failures, never by the first parallel pass — xdist-parallel runs on this box reclassify a rotating set of load-induced flakes as green on serial rerun; real failures persist. Reset (`stop-all` + `brutal_teardown.sh` + `pkill -x nginx`) between heavy runs — the box itself degrades under sustained repeated load (orphaned masters, climbing load average) in a way that makes diagnosis on a degraded box unreliable.

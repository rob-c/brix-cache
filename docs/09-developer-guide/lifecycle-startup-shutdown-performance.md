# Lifecycle startup & shutdown performance

**Status:** implemented 2026-06-27 · **Scope:** cold start, reload (SIGHUP),
worker respawn, full shutdown · **Related:** [reload-semantics.md](reload-semantics.md),
[postmortem-shmtx-semaphore-stall.md](postmortem-shmtx-semaphore-stall.md),
[optimizations.md](optimizations.md)

This is the engineering writeup of a measure-first pass over the module's process
lifecycle: how it was instrumented, what the numbers said, the one change that
mattered, and the things worth remembering. The companion design/results record is
`docs/superpowers/specs/2026-06-27-lifecycle-startup-shutdown-speed-design.md`.

---

## 1. The method: measure before touching anything

The brief was "make startup and shutdown faster," with no specific observed pain.
The right first move was **not** to start optimizing suspects — it was to make the
lifecycle observable, get honest numbers, and let the data choose the target. That
discipline paid off: the dominant cost turned out to be a single function nobody
would have guessed by eye, and three things that *looked* like plausible suspects
(SHM zone init, listen-socket setup, per-server-block preparation) were all
microseconds.

Two deliverables came out of Stage 1, by design:

1. **Permanent, low-overhead instrumentation** that ships in the module, so the
   breakdown is available on every boot and reload forever — not just during this
   investigation.
2. **A throwaway harness** for repeatable before/after measurement.

---

## 2. The instrumentation (permanent)

`src/compat/lifecycle_timing.{c,h}` is a tiny monotonic stopwatch. It accumulates a
sequence of named phase deltas into one fixed buffer and emits a single `NOTICE`
line per lifecycle event:

```
xrootd postconfig: prepare=2458us policy=1us registries=6us frm=0us pools_uring=13us total=2479us
xrootd init_process[w0]: uring=122us servers=85us keypool=17249us total=17458us
```

Design points that matter:

- **`CLOCK_MONOTONIC`, not `_COARSE`.** Coarse is ~1–4 ms granularity and would
  quantise sub-millisecond phases to 0/1000 µs. Monotonic is vDSO-backed (~20 ns a
  call) — the two reads per marked phase are lost in the syscalls each phase already
  makes. Microsecond integers, no floats (nginx avoids floats; it also keeps the
  output free of locale/formatting surprises).
- **One line per lifecycle event, never per request.** The accumulator is a stack
  struct owned by the caller — no allocation, no globals, nothing on the hot path.
- **`NOTICE` level** so the breakdown survives the default log level: operators get
  a boot-cost summary without raising verbosity, and it's greppable
  (`grep 'init_process\[' error.log`).

Wired into two hooks:

- **Master:** `ngx_stream_xrootd_postconfiguration` (`src/config/postconfiguration.c`)
  — buckets `prepare` (per-server prep + GSI/TLS/token/sss/krb5), `policy`,
  `registries` (metrics/dashboard/session/srv/tpc SHM zones), `frm`, `pools_uring`.
- **Per worker:** `ngx_stream_xrootd_init_process` (`src/config/process.c`) —
  buckets `uring`, `servers` (the per-server-block startup loop), `keypool`.

### Gotcha: `ngx_snprintf` does not null-terminate

The first version printed garbage (`pools_uring=16us 3�total=…`). `ngx_snprintf`
returns the new cursor but does **not** write a trailing `'\0'`, so passing the
buffer to a later `%s` reads past the written region. Every accumulator append must
terminate explicitly (`*pos = '\0'`) within the reserved capacity. This is a
recurring nginx-string trap — see [the FAQ in CLAUDE.md](../../CLAUDE.md) on
`ngx_str_t` not being null-terminated.

## 3. The throwaway harness

`tests/profile_lifecycle.sh` self-provisions a minimal instance (its own conf, a
self-signed cert, high ports, multiple workers) and measures the four events:

```
cold start (boot -> first accept) :     20 ms   # the real SLO
reload     (HUP  -> serving again):     14 ms
respawn    (kill -> worker back)  :     16 ms
shutdown   (quit -> master gone)  :     20 ms
```

plus the parsed phase breakdown and an optional `strace -c` syscall summary
(`STRACE=1`).

Two lessons baked into the harness:

- **Self-provision; don't lean on the shared fixtures.** `/tmp/xrd-test` is
  regenerated by `manage_test_servers.sh` and reaped by the environment's tmp
  cleaner — a profiler that assumes it exists will flake. The harness generates
  everything it needs each run.
- **Quiesce between measurements (`settle()`).** A reload spawns N new workers; if
  you snapshot the "new worker" counter for the *next* measurement before the
  previous event's workers have all logged their `init_process` line, you get a
  false ~0 ms reading. The fix is to wait until the worker count equals the target
  **and** the `init_process` line count has stopped climbing before each measurement.
  The first respawn number we saw (4 ms, vs the real ~16–31 ms) was this race.

## 4. What the numbers said

| phase | cost | note |
|---|---|---|
| **GSI keypool warm-up** (per worker) | **~16 ms** | **~97% of per-worker init** |
| master `prepare` (cert/TLS) | ~2–3 ms | distant second |
| GSI trust-store build alone | <100 µs (1 CA) | scales with CA-bundle size |
| SHM zones (`registries`) | ~6 µs | a non-issue |
| `uring`, `servers`, `policy`, `frm` | µs | non-issues |

The headline: **one synchronous loop generating 64 ephemeral Diffie-Hellman keys on
the event thread at worker start** (`xrootd_gsi_keypool_init`) was essentially the
entire per-worker startup cost, and the bulk of respawn time too. None of the
"obvious" suspects mattered.

This is exactly why measure-first matters. The keypool warm-up exists to keep keygen
*off* the event thread under a handshake burst ([Phase 33], see
`src/gsi/keypool.h`) — a deliberate, correct decision. But the *warm-up itself* ran
synchronously at boot, paying the whole cost up front on every worker start and
respawn.

---

## 5. The fix: lazy keypool seed (the one change that mattered)

The async refill machinery already existed for steady state (`xrootd_kp_schedule_refill`
posts a batch to the thread pool when the pool runs low). The warm-up just wasn't
using it.

```text
  worker init_process timeline  ── ▓ = on the event thread (blocks accept)
  ─────────────────────────────────────────────────────────────────────────
  BEFORE: 64 DH keys generated synchronously before the worker accepts
   uring│servers│▓▓▓▓▓▓▓▓▓▓ 64 keys ▓▓▓▓▓▓▓▓▓▓│ ──▶ accept   ~16–18 ms
                 └──────── ~16 ms, ~97% of init ────┘

  AFTER: seed a few, accept immediately, fill the rest off-thread
   uring│servers│▓ seed=4 │ ──▶ accept                       ~1.4 ms
                  └ ~1.2ms │
                          └····· thread pool fills 4→64 ·····▶ "warmed 64/64"
                            (off the event thread, no accept delay)

  ⚠ engages ONLY if a thread_pool is configured — else falls back to the
    synchronous full warm-up (correctness over latency, no speedup)
```

`xrootd_gsi_keypool_init(cycle, pool, target, seed)` now:

1. generates only `seed` keys (default **4**) **synchronously** — enough to serve
   the first few handshakes before the off-thread fill lands;
2. schedules the remainder up to `target` (default **64**) **off the event thread**
   via the GSI server's thread pool; `xrootd_kp_refill_done` chains one batch at a
   time until the target is reached;
3. falls back to a full synchronous warm-up **only when there is no thread pool** —
   correctness over latency, behaviour unchanged for those deployments.

Two new directives expose the knobs: `xrootd_gsi_keypool_size` (target) and
`xrootd_gsi_keypool_seed` (synchronous boot seed). The static ring is sized to a
compile-time ceiling `XROOTD_GSI_KEYPOOL_CAP` (256) so the directive can't be set
unboundedly high; the low-water that triggers steady-state refill is now `target/2`.
A one-shot `"GSI DH key pool warmed to N/N keys (off-thread)"` NOTICE confirms the
background fill completed.

### Result

| | before | after |
|---|---|---|
| per-worker `init_process` total | ~16–18 ms | **~1.4 ms** |
| keypool warm-up | ~16 ms | **~1.2 ms** (seed only) |
| worker respawn | ~28 ms | **~16 ms** |

A ~13× cut in the dominant per-worker startup cost, with the warm pool still
reaching full size moments later, off the event thread.

### The caveat that bit the harness — and matters in production

**The lazy path only engages when a thread pool is configured.** If neither
`thread_pool` nor `aio threads` is present, `ngx_thread_pool_get("default")` returns
NULL and the code correctly falls back to the synchronous full warm-up — so there is
**no speedup**. This nearly read as "the change does nothing": the first harness runs
still showed ~16 ms because the generated conf had no thread pool. Adding
`thread_pool default threads=4` made the win appear. GSI servers doing async file I/O
normally have a pool already, but it's worth stating in operations docs.

The tradeoff is honest and small: in a narrow window right after boot, an immediate
GSI handshake burst larger than `seed` could fall back to inline keygen until the
first off-thread batch lands. The `pop` path already handles an empty pool by
inline-generating and kicking a refill, so this is a brief latency blip, not a
failure. Tune `seed` up for handshake-burst-heavy front doors.

## 6. The master `prepare` cost: why we instrumented instead of "fixed" it

`prepare` (~2–3 ms) is the GSI `X509_STORE` build — `xrootd_rebuild_gsi_store`
parsing every CA and CRL under the trusted-CA path. Two facts shaped the decision:

- It **cannot be safely deferred**: the store must exist before the first GSI
  handshake. Deferring an auth-path trust object to "fill in later" trades a couple
  of milliseconds for a correctness/security hazard. Not worth it.
- Its cost **scales with CA-bundle size**. On a one-CA test config the store build is
  <100 µs and the ~2 ms is fixed RSA-key/TLS-context init; on a full grid CA
  distribution (hundreds of CAs + CRLs) the store build dominates GSI config time.

So rather than a risky deferral, the store build is now **independently timed**:

```
xrootd: GSI trust store built from "/etc/grid-security/certificates" in 41234us
```

An operator seeing a slow reload can attribute it at a glance. This is the
measure-first instinct again: when the safe optimization is small and the unsafe one
is tempting, make the cost *visible* and let the deployment that actually has a big
bundle decide. A genuinely safe future win exists — de-duplicating the store build
across server blocks that share a CA path within one config pass — but it's low
priority and carries `X509_STORE` lifetime/refcount risk, so it was left as a noted
follow-up rather than rushed.

---

## 7. Why shutdown and reload were left mostly alone

They were already fast (~20 ms / ~14 ms) thanks to prior fast-teardown work: timers
guarded by `ngx_exiting`, idiomatic idle-connection close via `c->idle` +
`ngx_close_idle_connections()`, clean-FIN retry signalling, and mid-transfer resume
across restart. See [reload-semantics.md](reload-semantics.md). The instrumentation
confirms they don't regress; there was no remaining cost worth chasing. Worth
remembering: nginx exits once `ngx_event_no_timers_left()` returns OK, so the
dominant lever on teardown speed is *stopping background timers*, not sweeping
connections — idle conns have no timer and never pin the worker.

## 8. Verification

- Build clean under `-Werror`; `nginx -t` passes with the new directives.
- **GSI auth path not regressed:** `test_gsi_handshake.py` + `test_gsi_concurrency.py`
  = **79 passed** against real stock `xrdfs`/`xrdcp` clients (the keypool exists for
  concurrency, so the concurrency suite is the load-bearing guard).
- **Feature guard:** `tests/test_lifecycle_speed.py` = **4 passed** — self-provisioning
  (no fleet dependency; run with `TEST_SKIP_SERVER_SETUP=1`), asserting the lazy path
  (small synchronous seed + off-thread fill to target), the synchronous fallback when
  no thread pool exists, configurable size, and well-formed phase lines.

## 9. Generalisable lessons

1. **Measure-first is not ceremony.** The dominant cost (a DH-keygen loop) was
   invisible to inspection and unrelated to every plausible suspect. One NOTICE line
   found it in minutes.
2. **The async machinery you need for steady state is often the same machinery you
   need at boot.** The keypool already knew how to fill off-thread; the warm-up just
   wasn't asking it to.
3. **Don't defer an auth-path object to save milliseconds.** When the safe
   optimization is small and the unsafe one is tempting, instrument the cost instead.
4. **A speedup gated on optional config can masquerade as a no-op.** State the
   precondition (here: a thread pool) loudly, in code comments, tests, and ops docs.
5. **Permanent, cheap instrumentation outlives the investigation.** The phase lines
   stay useful for every future boot, reload, and "why is startup slow on this host"
   ticket.

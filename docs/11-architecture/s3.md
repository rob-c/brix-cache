# S3 request lifecycle

How an S3-style HTTP request gets routed to the right handler: SigV4 auth, query-string dispatch, and multipart staging.

[← Architecture overview](index.md)

## S3 request lifecycle

S3 requests also use nginx's HTTP layer. The S3 module handles the AWS
path-style API subset used by XrdClS3 clients.

```
S3 client (XrdClS3, aws-cli, boto3)
    │  HTTP(S) request
    │  GET /bucket/key | PUT /bucket/key | POST /bucket/key?uploads | ...
    ▼
nginx HTTP (+ optional SSL)
    ▼
ngx_http_s3_handler()                  src/protocols/s3/handler.c
    │
    │  1. Parse request URI into bucket + key
    │
    │  2. Optional SigV4 auth check
    │      s3_verify_sigv4()            src/protocols/s3/auth_sigv4_verify.c
    │      if brix_s3_access_key set: verify AWS Signature v4
    │      if not configured: anonymous access allowed
    │
    │  3. Routing by method + query string
    │
    │  POST /bucket/key?uploads
    │      s3_handle_multipart_initiate()     src/protocols/s3/multipart_initiate.c
    │      creates staging dir .<key>.mpu-<id>/
    │      returns XML with UploadId
    │
    │  PUT /bucket/key?partNumber=N&uploadId=ID
    │      read body → s3_put_body_handler()  src/protocols/s3/put.c
    │      writes part to staging dir as part.<N>
    │      returns ETag header
    │
    │  POST /bucket/key?uploadId=ID
    │      read body → s3_multipart_complete_body_handler()  src/protocols/s3/multipart_complete_body.c
    │      parses XML part list
    │      assembles parts in ascending part-number order
    │      atomic rename temp → final path
    │      removes staging dir
    │
    │  DELETE /bucket/key?uploadId=ID
    │      s3_handle_multipart_abort()        src/protocols/s3/multipart_abort.c
    │      recursive unlink of staging dir
    │
    │  GET /bucket/key
    │      serve file with Content-Length and ETag
    │
    │  HEAD /bucket/key
    │      stat only — no body
    │
    │  PUT /bucket/key  (single-part)
    │      read body → s3_put_body_handler()  src/protocols/s3/put.c
    │      write to temp → atomic rename
    │
    │  DELETE /bucket/key
    │      unlink via confined helper
    │
    │  GET /bucket/?list-type=2
    │      s3_handle_list()                   src/protocols/s3/list_objects_v2.c
    │      opendir + readdir, XML response
    │
    ▼
  response sent via nginx HTTP machinery
```

### S3 multipart staging layout

```
/data/                              ← brix_export
  mybucket/                         ← bucket directory
    bigfile.dat                     ← final object (after Complete)
    .bigfile.dat.mpu-a1b2c3d4/     ← staging dir (hidden, prefixed with .)
      part.1                        ← first uploaded part
      part.2                        ← second uploaded part
      ...
      part.N                        ← last part
```

Part numbers are validated as decimal integers in the range 1–10,000 before
appearing in any filesystem path. All paths go through confined-open helpers to
prevent traversal. The staging directory is removed by `AbortMultipartUpload`
or after successful `CompleteMultipartUpload`.

### Key design decisions

- **No memory buffering for large uploads**: PUT and multipart UploadPart use
  nginx's body-spooling to a temp file, then `copy_file_range(2)` from temp to
  final location. Large files never fully buffer in memory.
- **Atomic assembly**: CompleteMultipartUpload writes to a temp file, then does
  a single `rename(2)`. Readers never see a partially assembled file.
- **Part number as path component**: validates strictly (1–10,000, decimal only)
  before constructing any path string.

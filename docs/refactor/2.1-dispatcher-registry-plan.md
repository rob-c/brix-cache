# Plan — Dispatcher consumes the protocol registry; migrate per-protocol handling

Follows QUALITY_ROADMAP 2.1. The registry seam already exists
(`src/protocols/common/protocol.{h,c}` — descriptor = name + module family).
This plan wires it into the live dispatch paths and migrates per-protocol logic
behind it, **without** inventing a false uniform open/read/write/close vtable
(root:// is a stream module; s3/webdav/cvmfs are http modules — two different
nginx request models).

## Guiding constraints
- Two families stay two families. The registry unifies *identity + policy*
  (name, family, enable-check, exclusivity), NOT request I/O.
- Behavior-preserving, incremental. Each phase builds green + passes the suite;
  land one phase per PR.
- Reuse, don't duplicate: `src/core/types/proto_list.h` stays the id/label truth;
  `src/protocols/shared/` stays the shared-code home. The registry references
  them, it does not replace them.

## Phase 0 — Enrich the descriptor (small, no behavior change)
Extend `brix_protocol_t` with the fields the dispatch/config layers actually key
off today, sourced from existing code:
- `brix_proto_t id;` (link to proto_list.h — enables `brix_protocol_by_id`).
- `ngx_flag_t (*enabled)(void *conf);` — per-protocol "is this location mine?"
  (webdav `common.enable`, s3 handler enable, cvmfs module gate, root stream).
- `int listener_port_default;` (observability/tests convenience).
Add `brix_protocol_by_id()`. Register descriptors with these populated.
*Verify:* build + `check_config_coverage`. No runtime consumer yet.

## Phase 1 — Registry drives protocol exclusivity
Target: `src/protocols/shared/proto_exclusive.c` ("one brix protocol per
location+port"). Replace its hand-maintained per-protocol set with a scan over
`brix_protocol_find`/the registry table (family-aware: a location may host one
http protocol; a stream server one stream protocol).
*Verify:* `proto_exclusive` unit/e2e + the multi-proto conformance tests.

## Phase 2 — HTTP dispatch keys off the registry (family = HTTP)

> **STATUS: satisfied-by-design / N/A (2026-07-09).** There is no central
> per-location branching to replace. Each http protocol already registers at its
> correct nginx seam — s3/cvmfs via `clcf->handler` (per-location content
> handler), webdav via a four-phase handler set (ACCESS/PRECONTENT/CONTENT/LOG) —
> and each self-guards on its enable flag. P1 makes the registry the single source
> that enforces one http protocol per location. A single `brix_protocol_http_dispatch(r)`
> would REGRESS webdav's multi-phase model and add a hot-path facade for no gain
> (same nginx-mismatch as the generic open/read/write/close vtable). Dispatch is
> therefore left idiomatic; the registry governs *identity + exclusivity*, not I/O.

The three http protocols each register a phase handler today. Introduce a thin
`brix_protocol_http_dispatch(r)` in `src/protocols/common/` that, at the http
access/content phase, uses the registry to pick the enabled http protocol for
the location (via each descriptor's `enabled` hook) and calls its existing
handler — replacing any ad-hoc per-location branching. Per-protocol handlers
(`s3/handler.c`, `webdav/dispatch.c`, `cvmfs/handler.c`) are UNCHANGED internally;
only *selection* moves to the registry.
*Verify:* s3 + webdav + cvmfs conformance suites; `run_suite.sh --pr`.

## Phase 3 — Stream side records identity via the registry
root:// keeps its opcode state machine (`protocols/root/connection/` +
`handshake/dispatch*`). No dispatch change; register its descriptor and route
identity/label/metrics lookups (proto name, family) through the registry so the
stream and http planes share ONE identity source. Fold `brix_protocol_register_all()`
into module init (postconfig) so the table is populated at startup.
*Verify:* root:// conformance + sesslog/metrics labels unchanged.

## Phase 4 — Consolidate shared request scaffolding (optional, incremental)

> **STATUS: substantially done; no further safe lift (2026-07-09).** The genuinely
> shared HTTP semantics are already extracted to `src/core/http/` (etag,
> conditionals, body, compress, file-response) and `tools/ci/check_http_helper_reimpl.sh`
> (green) enforces that no protocol regrows a private copy. The remaining
> per-protocol scaffolding is NOT real duplication: S3 CORS is per-bucket
> CORS-rule driven, WebDAV CORS is browser Access-Control-* — different semantics;
> and each protocol's method-Allow set is its own (S3 vs WebDAV's PROPFIND/MKCOL/
> COPY/… vs cvmfs GET/HEAD). Lifting those behind a shared hook would be a false
> merge. Left as-is; revisit only if a genuinely identical helper appears (the
> reimpl guard will flag it).

Where the http protocols genuinely duplicate scaffolding (CORS, method-allow,
range/conditional plumbing already partly in `src/core/http/` + `shared/`), lift
the common bits behind registry-selected hooks on the descriptor
(`allow_methods`, `cors`) — one helper at a time, each guarded by
`check_http_helper_reimpl`. Skip anything that isn't real duplication.
*Verify:* per-move build + targeted tests; no new `check_http_helper_reimpl` hits.

## Out of scope / do not do
- No unified open/read/write/close vtable across stream+http (false abstraction).
- No rewrite of the root opcode dispatcher or the http phase model.
- No merging of `proto_list.h` into the registry (keep id/label generation there).

## Effort / sequencing
P0 ~½d · P1 ~1d · P2 ~2–3d · P3 ~1d · P4 open-ended (per-helper). Land P0→P3 in
order; P4 opportunistically. Each phase: build green + suite green + one commit.

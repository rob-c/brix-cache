Frequently asserted doubts

Good questions.
Here are the receipts.

Every new implementation of a 20-year-old protocol earns skepticism, and it should. This page collects the hardest objections we hear — architectural, performance, security, and "this will never last" — and answers each one with a citation into the repository. The rule is the same as everywhere else on this site: nothing aspirational, and the what-it-doesn't-do section is load-bearing.

Objection class 1

“nginx is a web server.”

“root:// is a single long-lived TCP connection multiplexing many requests. nginx serves short web requests. This can’t work.”

Two different nginx engines. BriX-Cache's native root:// does not live in the HTTP engine at all — it lives in the stream {} engine, the part of nginx built to own raw, long-lived TCP and TLS connections. The module owns the socket from accept, frames kXR requests off its own receive loop, and demultiplexes by streamid exactly as the protocol requires. A long-lived connection is the easy case for an event loop: an open-but-idle connection costs kilobytes of state, not a parked thread. And this is not a translation shim onto HTTP — it is the XRootD 5.2 wire protocol implemented natively: 33 opcodes recognized, 32 active, in-protocol TLS upgrade, kXR_sigver request signing.

src/protocols/root/ · docs/05-operations/operation-status.md

“An event loop stalls the moment it touches a disk. Storage servers need threads.”

They need threads for I/O, not for connections — and nginx has them. Every data-plane disk syscall (read, pgread, readv, write, pgwrite, WebDAV PUT) is dispatched to an nginx thread pool while the worker keeps framing protocol and servicing other sockets; one blocking read cannot stall ten thousand connections. Cleartext reads go out via file-backed sendfile; TLS paths use memory-backed buffers — never mixed. The difference from thread-per-connection is that the thread count follows concurrent disk operations, not open sockets.

thread_pool brix_pool threads=8 · docs/03-configuration/rpm-package-build.md

“It won’t do the hard parts of the protocol — pgread/pgwrite, request signing, TLS upgrade.”

kXR_pgread / kXR_pgwrite carry a CRC32c per 4K page, aligned to absolute file offset, so corruption is caught per page — including the pgwrite checksum-error accept-then-correct retransmit exchange. kXR_sigver HMAC-SHA256 request signing is implemented for GSI sessions, and TLS is negotiated in-protocol the way the 5.x spec describes. These paths run under differential conformance tests against the reference implementation, not just against our own client.

src/protocols/root/read/pgread.c · src/fs/backend/csi_verify.c

“nginx workers are separate processes. Grid storage needs shared state — you can’t coordinate open files, caches, and TPC across workers.”

Shared memory, done carefully. Cross-worker state — open-file registries, cache locks, the native-TPC rendezvous, rate limiters — lives in SHM tables created through one audited allocation helper. That helper exists because we hit and published the failure modes: a lost semaphore wakeup in stock shared-memory mutexes produced 60–450 s stalls on the hot open path, so the repo enforces a spin-and-yield mutex rule codebase-wide, plus a full audit of every lock a dying worker could strand — each class pinned by its own regression test.

docs/09-developer-guide/postmortem-shmtx-semaphore-stall.md · tests/test_shm_mutex_recovery.py

“A plugin bolted onto a web server will sprawl. Storage code will leak into every corner.”

The opposite is mechanically enforced. All bytes, namespace, and metadata funnel through one VFS; raw data-plane syscalls are permitted in exactly one directory, src/fs/backend/, behind a capability-typed driver vtable. A CI guard fails the build if any protocol handler reaches around the seam — the migration to that rule was tracked as a monotone counter, 56 violations down to 0, and it is not allowed to grow back. That is why POSIX, block-store, and S3 backends sit under the same protocol front-ends without the front-ends knowing.

tools/ci/check_vfs_seam.sh · docs/11-architecture/overview.md

“So it’s really just XRootD-flavoured HTTP underneath.”

No — it is one binary with two native fronts. The stream {} block speaks the binary kXR protocol; the http {} block speaks WebDAV, S3, and metrics. Both converge on the same auth layer, the same path confinement, the same VFS, the same cache — written once. The same physical file is reachable as root://, davs://, and s3:// simultaneously because the protocols are peers, not layers.

src/protocols/{root,webdav,s3}/ · docs/11-architecture/overview.md

“You can’t upgrade a live connection to TLS mid-stream inside nginx. The 5.x in-protocol TLS handshake is impossible here.”

It's implemented, and it's load-bearing. The client negotiates TLS in-protocol exactly as the 5.x spec describes — roots:// from byte one, or cleartext handshake with an in-protocol upgrade — and the codebase holds a strict invariant on the other side of that switch: TLS connections use memory-backed buffers, cleartext connections use file-backed sendfile, and the two are never mixed on one path. That invariant is written down, enforced in review, and exercised by the conformance suites over both transports.

src/protocols/root/handshake/ · src/net/upstream/tls.c

“Config reloads on a web server mean dropped transfers. Storage can’t tolerate that.”

Reloads are nginx's party trick, not its weakness: nginx -t refuses a broken config before a single byte is accepted (missing certs, JWKS, CRLs all fail closed), and a reload starts new workers while old workers drain — in-flight transfers finish on the old generation. One honest caveat, tracked in the gap register: the WebDAV lock table does not yet persist across a reload. A fast-teardown-during-reload bug that could lose one in-flight streaming read was found by the suite and fixed, with the postmortem in the tree.

docs/09-developer-guide/reload-semantics.md · docs/10-reference/feature-gap-analysis.md

“sendfile is an HTTP trick. You can’t zero-copy a framed binary protocol.”

You can when the framing allows it, and only then. Cleartext root:// reads take a file-backed sendfile fast path when the handle is a plain file; driver-backed and multi-block handles take the buffered positional-read path. We know the boundary is right because we got it wrong once — an early fast path could truncate a multi-block read at the first block — and the bug, the fix, and the gating rule are documented rather than buried.

docs/09-developer-guide/history-storage-and-caching.md §sendfile

“A standalone server is a toy. Real deployments are federated — redirectors, clusters, upstream proxies.”

The scale-out plumbing is its own subsystem: src/net/ carries the cluster/redirect integration, upstream and proxy connectors, and rate limiting. And because redirect topologies are an attack surface, they were threat-modeled early: redirect poisoning is countered with sss message protection, CIDR restriction, and host validation as part of the phase-28 threat-model work — hardening that landed with the feature, not after an incident.

src/net/ · docs/07-security/threat-model.md

“Proxied writes are where data quietly dies. A buffering middlebox will ack bytes it never stored.”

Every backend write path converges on one verified-writer object — open, write, commit, abort — for POSIX, block-store, and S3 alike. Commit folds two checks: a self-computed rolling checksum accumulated as bytes arrive (order-independent, so out-of-order parallel streams converge; gaps and overlaps fail closed) and a read-back verification against what storage actually kept. On mismatch the object is unlinked and the client gets an error — a failed transfer, never a silently corrupt file wearing a success code.

src/fs/vfs/vfs_writer.c · src/core/compat/wverify.c

Objection class 2

“It will be slow.”

“A reimplementation is always slower than the original.”

Measured, not assumed — and measured fairly. With both servers configured identically (TLS off on both data paths, same sink), a single-stream 1 GiB root:// read runs at ~297 MiB/s through BriX-Cache vs ~242 MiB/s through the reference daemon on the same host — parity to modestly ahead, single stream, POSIX backend. Loopback HTTP GET reaches ~1.9–2.2 GB/s. We publish these as relative, same-host comparisons, not absolute hardware claims.

docs/05-operations/performance-benchmarks.md · README.md

“One thread per connection has served XRootD for 20 years. Why would an event loop matter?”

Because concurrency is where thread-per-connection bends. Aggregate throughput of a thread-per-connection design degrades as connection counts climb; an event-driven worker model holds its throughput as connections scale, which is the entire reason nginx displaced thread-per-connection web servers. The reference daemon also carries a fixed floor of roughly 16 infrastructure threads per instance regardless of configuration — and its HTTP bridge routes every request through the xroot scheduler, so a small scheduler pool can deadlock HTTP entirely. An idle BriX-Cache connection is a few kilobytes of worker state.

docs/05-operations/performance-benchmarks.md · TESTING.md §xrd.sched

“Fine on a LAN. Real WANs lose packets, and then it will fall apart.”

This is where the gap is widest — in the other direction. Under 1% packet loss, BriX-Cache root:// transfers stay byte-exact at ~107 MB/s (~660 tuned); the reference client-server pair drops to ~2.2 MB/s with 15–45 second stalls on the same emulated link — roughly a thousand-fold difference in loss tolerance. Over HTTP under the same loss, BriX-Cache serves byte-exact via Range-resume. The fault lane exists precisely to keep this true: a TCP fault proxy resets and stalls connections mid-read while tests assert byte-exact results and zero application-visible errors.

README.md §Performance & resilience · tests/c/fault_proxy.c

“Vendor benchmarks are marketing. Why should anyone trust these?”

Because the methodology corrections are published alongside the numbers. The harness originally compared nginx-with-TLS against xrootd-with-cleartext — flattering to nobody — and the fix, the fair re-run, and the conclusion are written down, including the part that doesn't flatter us: single-stream speed alone does not justify the architecture; the case comes from concurrency and resilience. We also document that kernel-TLS measured slower on our hardware rather than shipping it as a checkbox feature. Every number carries its conditions.

docs/09-developer-guide/history-testing-and-incidents.md §fair-comparison

“For caching, squid and varnish are solved problems. Why reinvent them?”

We didn't argue with them — we benchmarked against them. The in-tree cache matrix runs the same workload across module-reverse, module-proxy, stock nginx, squid, and varnish, and the baselines are checked into the repo (squid cold TTFB p50 7.09 ms, varnish 4.35 ms; zero errors, zero corrupt objects, zero stampedes for all). What the built-in cache adds is what a generic HTTP cache cannot: it sits behind the same grid auth, coalesces a client storm into a single upstream fill, serves stale-if-error when every origin is down, and verifies content hashes on arrival.

baseline_squid.json · baseline_varnish.json · tests/cmdscripts/cvmfs_matrix.py

“XCache already exists for this.”

XCache is a separate deployment wearing the thread-per-connection architecture; here the cache is a backend inside the same event-driven server, behind the same auth and the same VFS seam. Its failure modes have been hunted the hard way: a stuck origin is detected in seconds (2 s connect / 4 s stall defaults) and retried on fresh connections; a cache-fill lock stranded by a dead worker is reclaimed instead of returning 300-second kXR_FileLocked stalls — that one was found, fixed, and pinned with its own regression test.

src/fs/backend/cache/ · tests/test_cache_lock_reclaim.py

“What about metadata-heavy workloads? Small-file physics analysis is all stat and open.”

Two honest numbers. On the POSIX backend, metadata operations are native syscalls behind the confined resolver — nginx adds no daemon round-trip. The block-store backend pays a measured ~2.5–2.8× metadata penalty against POSIX (a documented write-ahead-log commit floor, ~54% of CPU in sqlite), which is why it's a per-export choice, not a default. The repo ships its own metadata benchmark (metabench) so you can measure your workload rather than trust ours.

docs/09-developer-guide/history-storage-and-caching.md · client/lib/observability/metabench/

“POSIX access through FUSE is always slow and fragile.”

Fragile is testable, so we test it: the xrootdfs FUSE resilience suites assert byte-exact reads with zero EIO surfaced to applications while a fault proxy resets and stalls the underlying link mid-transfer. For workloads that can't take FUSE at all, the client suite also ships an LD_PRELOAD POSIX shim and brixMount, a CVMFS-free writable overlay — three mount strategies, one protocol stack.

tests/test_xrootdfs_resilience.py · client/apps/fs/

Objection class 3

“It can’t be trusted.”

“A web server can’t do grid security. GSI, VOMS, proxy certificates — that’s decades of specialist code.”

All implemented, natively: GSI/X.509 with RFC 3820 proxy-chain validation and VOMS attribute extraction, kXR_sigver signing, WLCG JWT bearer tokens (JWKS, storage.read/write/create/modify scope enforcement), sss shared-secret, krb5, pwd, host auth, macaroon issue and verify, and OCSP with stapling. The clean-room client suite ships the other half — its own xrdcp/xrdfs with GSI — with no dependency on upstream libXrdSec.

src/auth/ · docs/05-operations/operation-status.md

“A reimplementation will quietly accept things the battle-tested original rejects.”

We tested exactly that, differentially, against XRootD v6.1.0 — and found the arrow points the other way. Stock XRootD does not enforce Globus signing-policy files, does not enforce RFC 3820 limited-proxy monotonicity, only warns on expired CRLs, and XrdHttp's TLS layer performs no GSI chain or signing-policy verification at all. The differential run found three concrete cases where stock accepted — out-of-namespace certs, wrong-CA-policy certs, revoked certs — and BriX-Cache rejects. The recorded conclusion: stricter than stock in every area where we differ.

docs/09-developer-guide/history-testing-and-incidents.md §WLCG-x509 differential

“One crafted path and it serves /etc/passwd.”

Path confinement is a kernel-enforced invariant, not a habit. Every wire path goes through one canonical resolver before any open, and confined opens use openat2(RESOLVE_BENEATH | RESOLVE_NO_MAGICLINKS) — a hostile symlink cannot walk out of the export even if a check were missed above. When an audit found an S3 UploadPartCopy path that bypassed the resolver, it was fixed by routing it through the same confined-open helper — and that class of bug is exactly what the single-seam CI guard exists to prevent recurring.

src/fs/path/beneath.c · tools/ci/check_vfs_seam.sh

“Multi-user means it runs as root, and an nginx module running as root is a nightmare.”

Workers are standard unprivileged nginx workers. Per-request UNIX identity is applied through a small privileged broker process holding only CAP_SETUID/SETGID, double-forked away from worker memory — and that boundary has been through nine adversarial red-team rounds (224 growing to 3,842 checks) without a breach. Backend credentials go further: BriX-Cache authenticates to origins as the requesting user (per-identity GSI proxy, bearer token, SigV4 key, or CephX keyring), and deny-mode guarantees a request without a mapped credential fails before any origin operation can ride the service account.

docs/09-developer-guide/multi-user-backend-credentials-through-the-vfs.md

“New codebase, new CVEs. It just hasn’t been attacked yet.”

It is attacked continuously — by its own repo. The suite carries hostile-actor lanes (evil_actor v1–v3, hostile token forge, evil paths, oversized frames, fd-exhaustion), a fault-injection proxy, and a multi-user authorization conformance matrix whose invariant — cached verdict must equal cold verdict — caught six rounds of real cross-user leaks before any release. Vulnerabilities found are fixed and published: an auth fail-open on the proxy branch, an sss deny-path flaw, missing macaroon expiry, a non-CSPRNG session id. A project that documents its own holes is a safer bet than one that hasn't looked.

docs/09-developer-guide/history-security-and-credentials.md · tests/tokenforge.py

“Everyone claims WLCG token support. It’s always half-baked.”

Ours came with an adversary attached. Token conformance is 103 wire and unit tests plus a hostile token forge that manufactures malformed, expired, mis-scoped, and mis-signed tokens against every endpoint. The forge earned its keep: it found that WebDAV GET, HEAD, and PROPFIND skipped scope checking — a token with no read scope could read — and that hole was closed before release. Scopes (storage.read/write/create/modify) are enforced per-operation, JWKS refresh included, and a JWT-expiry edge case (exp only enforced when positive) is on the published fixed list.

tests/tokenforge.py · docs/09-developer-guide/history-security-and-credentials.md

“Third-party copy is where credential handling goes to die.”

It's where ours is most explicit. HTTP-TPC supports OIDC with RFC 8693 token-exchange delegation; GSI delegation uses the standard GridSite two-step capture; and a delegated proxy is accepted only after full cryptographic chain verification against the CA — never by matching a DN string. On the S3 edge, SigV4 identities can mint short-lived x509 credentials for onward grid hops, so a bucket user's transfer never rides a shared service certificate.

src/protocols/webdav/tpc_cred.c · docs/09-developer-guide/history-security-and-credentials.md

“One hostile user and the whole box is unusable. Web servers don’t understand grid DoS.”

This one understands identities, not just IPs. Rate limiting keys on the authenticated principal — stock XrdThrottle keys on username only — with per-principal concurrency caps from the threat-model work. Below that: oversized frames are refused at the framing layer, fd exhaustion is tested (hundreds of data-channel opens without a leak), and the tree ships fail2ban filters and jails for the ban-hammer layer. And when the suite found the rate limiter itself misbehaving — a drain bug serving ~7 req/s where 30 were configured — that became a documented fix, not a footnote.

src/net/ratelimit/ · deploy/fail2ban/

“Attackers will poison your logs and blow up your metrics cardinality mid-incident.”

Both are closed by construction. Every string that arrives over the wire is sanitized before it reaches a log line, so terminal-escape and log-injection games die at the boundary. Metric labels are fixed and low-cardinality — no per-path, per-user, or per-token label explosion — so the Prometheus endpoint stays cheap and standing precisely when you need it most: during the incident.

src/observability/accesslog/access_log.c · brix_sanitize_log_string

“A directory full of delegated user credentials is a honeypot.”

It's treated as one. The per-identity credential store is reached through a single checkpoint function; mapping keys are strictly validated (a crafted identity of .. attempting path traversal is rejected — found by audit, fixed, regression-pinned); credential expiry is checked at time of use, not time of storage; and in deny mode a permanently lapsed credential dead-letters the pending write-back rather than ever falling back to the service account. The store's threat model is written down next to its design.

The scoreboard

Shipping here. Absent, late, or declined there.

"Where does this actually go beyond stock XRootD?" These rows. Each states what BriX-Cache ships today and what the reference platform does today — sourced from its own code and docs, so bring the receipts when you disagree. The full list of capabilities with no stock equivalent has its own page.

Per-user credential forwarding

brix-cache The proxy authenticates to backends as the requesting user — GSI proxy, WLCG bearer, S3 SigV4, or CephX, per identity, with a fail-closed deny mode.

stock xrootd XrdPssConfig.cc, in the source, today: “We don't support credential forwarding, yet.” The proxy layer rides a single service credential.

S3 — as a server

brix-cache A native SigV4 server endpoint (~6,900 LoC): GET, HEAD, PUT, DELETE, ListObjectsV2, multipart — the same files, over s3://.

stock xrootd S3 exists as a client/backend only. There is no stock way to serve your namespace to aws-cli or boto3.

WebDAV beyond class 1

brix-cache Class-2 LOCK / UNLOCK and PROPPATCH implemented, alongside full HTTP third-party-copy.

stock xrootd XrdHttp advertises DAV class 1. No locks, no property writes.

Monitoring without a decoder ring

brix-cache Prometheus-native metrics, a built-in dashboard, and WLCG Storage Resource Reporting over plain HTTPS/JSON.

stock xrootd Binary UDP f-stream/g-stream monitoring that needs a separate collector stack to become legible.

CRC64 checksums

brix-cache crc64 (CRC-64/XZ) and crc64nvme calculators built in, alongside adler32, crc32c, md5, sha1, sha256.

stock xrootd No CRC64 calculator ships with stock XRootD.

Macaroons & OCSP

brix-cache Macaroon issue and verify; OCSP verification with stapling.

stock xrootd Macaroon support is verify-oriented; OCSP stapling is absent.

Identity-aware rate limiting

brix-cache Limits keyed on the authenticated identity, with per-principal concurrency caps.

stock xrootd XrdThrottle keys on username only.

A GridFTP door, in 2026

brix-cache A maintained gsiftp gateway — GSI control and data channels, MODE E parallel streams, TPC — for sites still bridging ARC-CE and legacy fabric.

stock xrootd The Globus toolkit is end-of-life; the ecosystem answer is “migrate.” Sites that can't yet are on their own.

The honest column

What it doesn’t do.

“Is this a drop-in replacement for an XRootD deployment?”

No, and the docs say so in bold: this is “a high-quality reimplementation of XRootD protocols, not a drop-in replacement for the XRootD platform.” The deliberate non-goal is the tape/MSS ecosystem: migrate/purge orchestration is delegated to the systems that do it well (CTA, dCache, HPSS) — the built-in purge-watermark monitor observes and logs but never acts. If your site is an XrdFrm tape frontier, stock XRootD remains the right tool.

docs/09-developer-guide/history-protocols-and-feature-phases.md §13 · feature-gap-analysis.md

“What is currently broken or missing? Be specific.”

The open list is public. Native root:// TPC against a stock push-model (ofs.tpc) source does not work — nginx↔nginx and pull-capable sources (EOS, dCache) do. A hybrid mesh relaying a stock XrdHttp backend has an un-root-caused intermittent crash under load. The root:// write path to an S3 backend is still being wired. The gap register lists tape dispatch, outbound TLS-upgrade auth, multi-hop GSI delegation, lock persistence across reload, and third-party macaroon interop as known gaps with priorities. Skeptics are invited to check the register against our claims — that's what it's for.

docs/10-reference/feature-gap-analysis.md · docs/09-developer-guide/history-*.md

“Those benchmark numbers will not reproduce on my hardware.”

Correct — and we say so. Published figures are same-host, loopback, tmpfs-backed relative comparisons; their value is the ratio between the two servers under identical conditions, not the absolute MB/s. The benchmark harness and its configurations ship in the tree, so the comparison that matters — on your hardware, against your baseline — is one you can run yourself.

docs/05-operations/performance-benchmarks.md

“Where does it deliberately diverge from stock behavior?”

Two known divergences, both documented as decisions. On a kXR_pgwrite checksum error, stock retransmits the corrupt page; BriX-Cache hard-fails the request — a stricter posture, a behavior divergence, and not a data-loss risk either way. And parallel write streams (kXR_bind multiplexed writes) sit in the gap register at low priority: bind itself is implemented — its session id generation was hardened to a CSPRNG after an internal finding — but multi-stream write scheduling is not. The differential suites carry a clause-indexed decisions register for every case where we chose to differ.

tests/clauses/_decisions.py · docs/10-reference/feature-gap-analysis.md

“I heard there’s an unsolved crash. Doesn’t that sink the stability story?”

There is one, it's public, and it's bounded: a hybrid two-tier mesh in which BriX-Cache proxies a stock XrdHttp backend intermittently heap-corrupts under load (~8/100 requests surface as 502) and is not yet root-caused. Standalone operation, nginx↔nginx meshes, and every other proxy topology are unaffected. We'd rather ship a named open bug than a quiet workaround — that's the same discipline that produced the postmortems the stability story rests on.

docs/09-developer-guide/history-security-and-credentials.md §hybrid-mesh

Objection class 4

“It won’t survive.”

“Nobody tests a young storage server properly.”

The suite collects 9,764 tests, ~1,770 of which exist specifically to hurt the software — fault injection, chaos, resilience, hostile actors. Conformance is differential: whole test modules run unchanged against BriX-Cache and the reference daemon, over native root:// and WebDAV, using real libXrdCl, gfal, davix, and globus clients. That process cuts both ways — it caught a genuine data-loss bug in our own kXR_rm handling before release, and it caught stock accepting revoked certificates. Differential testing keeps everyone honest.

TESTING.md · tests/official_interop_lib.py

“A big C codebase in someone else’s server will rot.”

The architecture is guarded, not aspirational: seven concept buckets, one storage seam, five standing CI guards (VFS seam, config coverage, helper reimplementation, file-size ratchet, storage-driver conformance), and a tree that compiles -Werror with full RELRO and immediate binding. nginx itself is the least risky dependency in the stack — the module tracks stock nginx and loads as standard dynamic modules, no fork.

tools/ci/check_*.sh · docs/09-developer-guide/coding-standards.md

“Can I actually deploy it, or is this a laptop project?”

RPMs build in containers for AlmaLinux 8, 9, 10, and 11, producing module, client, tools, and test packages with an optional SELinux subpackage; runtime dependencies resolve from the standard WLCG repos. systemd units ship in the tree, config reloads drain gracefully, and a Kubernetes lab — Helm charts, in-cluster builds, and a remote suite that runs the full ~390-file test set from a client pod — is part of CI, not an afterthought.

packaging/rpm/ · k8s-tests/

“Why not just contribute all this to XRootD?”

Because the premise is architectural, not a patch. Event-driven workers, one mechanically enforced storage seam, and protocol front-ends as peers over a shared VFS are not features you can merge into a thread-per-connection framework — they are the frame. The two implementations interoperate on the wire, the differential suites test against stock continuously, and bugs found in stock during that testing are the community's to use. Two independent implementations of one protocol is how protocols stay honest — ask HTTP.

docs/11-architecture/overview.md

“FTS, Rucio, and gfal were built against the reference stack. They’ll break against this.”

That contract is tested with the real libraries, not simulated: a dedicated conformance batch (~1,160 tests) drives genuine libXrdCl and gfal bindings out-of-process against both servers and diffs every response. It surfaced nine wire-level divergences — down to the stat id field's inode/device encoding and kXR_ItExists error-code mapping — and all nine were fixed on our side. davix and the XrdHttp endpoint get the same differential treatment, and the gridftp door is exercised with globus-family clients.

tests/_xrdcl_proxy.py · tests/official_interop_lib.py

“AGPL is toxic. Our lab can’t touch it.”

Running BriX-Cache — at any scale, for any site — carries no obligation at all. The AGPL clause people fear applies only if you modify the software and serve users with the modified version: then you share the modifications. For publicly funded scientific infrastructure, that is the funding model stated as a license — improvements paid for by one agency's grant don't disappear into a private fork. If your lab consumes upstream releases, AGPL and Apache are operationally identical.

LICENSE · AGPL-3.0-only

“Bus factor of one. When the author gets bored, you’re stranded.”

The mitigation is that nothing load-bearing lives in anyone's head. The invariants are enforced by CI guards, not by memory; the development history, every postmortem, the coding standard, and an operation-to-file map for the entire protocol surface are in the tree; and the differential suites are an executable specification — a new maintainer who breaks compatibility finds out from the harness, not from users. Compare the usual alternative: institutional knowledge distributed across two decades of mailing-list archaeology.

docs/09-developer-guide/development-history.md · docs/09-developer-guide/coding-standards.md

“Enterprise EL deployments will fight it — SELinux, package deps, VOMS library soup.”

That fight happened in packaging, so it doesn't happen on your node. RPMs build reproducibly in containers for AlmaLinux 8 through 11; an optional SELinux subpackage labels /etc/grid-security correctly instead of telling you to setenforce 0; runtime dependencies resolve from the standard WLCG repositories; and the VOMS dependency is declared by soname, so the same spec installs against EL8's voms-libs and EL9's voms without an if-else in your config management.

For infra & OSS engineers

Three protocols, one core, no monolith.

BriX-Cache is a single nginx module. Three protocol front-ends parse the wire, one auth layer decides access, and all storage funnels through a VFS — raw file I/O exists in exactly one place.

The architecture, honestly

Snap-together parts, one seam for bytes.

shared-core

One event loop, three fronts

The stream block owns native root:// / roots://; http owns WebDAV, S3, and metrics. Both converge on one shared module core before touching a file — so auth, path confinement, caching, and metrics are written once.

vfs

The VFS is the sole storage truth

Bytes, namespace, and metadata all route through src/fs/. Raw pread / pwrite / sendfile live only in src/fs/backend/ behind a capability-typed driver — so an object or S3 backend can become primary without changing anything above it.

async

No worker ever blocks

Every disk syscall is handed to an nginx thread pool while the worker keeps framing protocol and servicing other sockets. Cleartext reads use file-backed sendfile; TLS uses memory-backed buffers — never mixed.

clean-room

A clean-room client suite

The repo ships its own xrdcp, xrdfs, diagnostics, checksum tools, a xrootdfs FUSE mount, a POSIX preload shim, and the C library libxrdc — built on the same in-tree protocol vocabulary, with no dependency on upstream libXrdCl / libXrdSec.

conformance

Tested against the reference

Whole test modules run unchanged against both BriX-Cache and the reference xrootd daemon — native root:// and the XrdHttp WebDAV endpoint. Any divergence in status code or bytes is a conformance failure, not a footnote.

wire

The full 5.2 wire protocol

All 32 active XRootD opcodes — open, read, pgread, readv, write, pgwrite, stat, fattr, sigver, and the rest — with per-page CRC32c on the paged paths.

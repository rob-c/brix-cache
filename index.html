Modular WLCG storage interface

The whole HEP data
stack in one binary.

BriX-Cache adds root:// (XRootD), WebDAV, and an S3 endpoint — and terminates a gsiftp:// GridFTP server in the same binary. Combine the parts that fit your site and keep one shared path for authentication, storage, caching, and metrics.

How it plugs in

Four fronts in, one path to disk.

Every protocol enters through nginx's event loop, funnels through one shared core that handles auth, path confinement, caching, and metrics — then reaches storage. No worker ever blocks: disk syscalls run on a thread pool while the worker keeps serving.

  1. clients xrdcp · curl · aws-cli
    →
  2. nginx event loop + module epoll · non-blocking workers
    →
  3. shared core auth · path-confine · cache · metrics
    →
  4. POSIX tree / root:// backend thread-pool I/O
Authentication

WLCG-native, checked before a byte moves.

GSI / x509 proxy certs with kXR_sigver request signing, VOMS, and WLCG bearer tokens with per-path scope enforcement (storage.read / write / create). Plus SSS, host allowlist, password, and Kerberos 5.

Observability

Every request is a metric and a log line.

Prometheus /metrics with fixed, low-cardinality labels — no per-file or per-user explosion. Structured access logs for every protocol, and a live HTTPS operator dashboard at /brix/.

brix_io_bytes_read{proto="webdav"} 9.2e11
brix_auth_total{method="gsi",status="ok"} 4201

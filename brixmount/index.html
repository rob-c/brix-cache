The threat model

What a bad network actually does.

Hotel wifi, an overloaded site uplink, a misbehaving inline firewall, a mirror mid-meltdown — from a mount's point of view they are all the same adversary. brixMount assumes all of it, all the time.

01

Resets connections mid-read

A firewall RST-injects 40 GB into your 60 GB file.

02

Stalls without failing

The TCP handshake completes, then the link black-holes bytes. No error, ever.

03

Mirrors go dark

The nearest replica stops answering ten minutes into your job.

04

Corrupts what arrives

Bytes land, but they are not the bytes the repository published.

05

Tampers in transit

A hostile middlebox serves you a catalog the project never signed.

06

Disappears entirely

The uplink dies. Your batch farm does not stop asking for files.

The defenses

One answer per threat. All of them landed code.

01 · resets

Reads that outlive the connection

A dropped or reset link triggers reconnect, full re-authentication, handle reopen, and offset resume — inside one patience budget (--max-stall), with bounded exponential backoff and jitter so a thousand clients don't re-hammer a flapping link in lockstep. The read completes; the application never sees it.

02 · stalls

Fail in seconds, not minutes

A connect ceiling caps the whole bring-up — TCP, protocol handshake, TLS, login — so a black-holed connection becomes a prompt, retryable failure instead of a hung mount. A separate per-operation I/O cap patrols steady state, because a stalled handshake and a legitimate large read deserve very different patience.

03 · dead mirrors

Failover with a memory

Replica and proxy failover with a timed blacklist: first failure earns a short probation, repeat offenders back off to a capped interval, recovered endpoints snap back on success. When a mirror dies mid-job, the next fetch has already routed around it.

04 · corruption

Every object hash-verified

Content comes from a content-addressed store, and every fetched object is verified against its hash on arrival. A damaged transfer, a truncated resume, a byte-flipping middlebox — none of it can go undetected, which is also what makes retrying against a different mirror mid-file safe.

05 · tampering

Trust is verified, not assumed

CVMFS manifests are accepted only through the full trust chain — repository whitelist, certificate fingerprint, manifest signature. A middlebox can drop your packets, but it can't hand you a catalog the repository never signed.

06 · outages

Offline is a mode, not an error

When every replica and proxy is blacklisted, brixMount drops to cache-only serving from its local content-addressed cache, and a failed catalog refresh keeps the current catalog serving. Total outage upstream; the mount keeps answering.

The proof

Torture-tested, not hoped-for.

Every claim above is enforced by tests, not marketing. The FUSE suites run through a TCP fault-injection proxy (fault_proxy.c) that resets connections mid-read and injects stalls and latency on command — and the tests assert byte-exact recovery with no EIO ever surfaced to the application.

Around it: dedicated CVMFS failover and upstream-stall suites, a netem network-emulation lab, and compressed-read fault coverage — inside a ~8,700-test suite whose slow lane (~1,770 tests) is devoted to resilience, chaos, and fault injection.

What you're mounting

One command. Five mount types.

cvmfs

CVMFS-brix

Native CVMFS repositories as a read-only mount: SQLite catalogs, content-addressed fetch, the full signature trust chain, and a shared local cache. autofs on-demand mounting under /cvmfs/<repo> works stock-style.

cvmfs-rw

CVMFS-brix, writable

The same repository plus a local writable overlay: create, edit, and delete on top of read-only software stacks. Your changes live in a plain, inspectable directory — list them with --overlay-list, discard with --overlay-reset.

eos · root · roots

XRootDFS-brix

Any XRootD or EOS endpoint as a POSIX directory, over root:// or TLS roots:// — with the same reconnect-and-resume machinery underneath every read.

$ brixMount cvmfs atlas.cern.ch ~/atlas # read-only repo
$ brixMount cvmfs-rw atlas.cern.ch ~/atlas # + writable overlay
$ brixMount eos root://eoslhcb.cern.ch ~/eos # XRootD/EOS backend
$ brixMount --overlay-list ~/atlas # what have I changed?

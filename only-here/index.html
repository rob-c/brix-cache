Distinctive capabilities

More options for your storage service.

This page highlights capabilities that BriX-Cache adds beyond a stock XRootD deployment. Each card explains the capability, describes the reference behavior, and links it to the implementation. For features shared by both projects, see the FAQ comparison.

Distinctive capability · identity

Requests keep their owner, end to end.

Identity stays with the request across the proxy hop. Each backend operation can use the requesting user’s credential instead of a shared service identity.

Per-user backend credential forwarding

The server authenticates to origins and backends as the requesting user — a per-identity GSI proxy, WLCG bearer token, S3 SigV4 key, or CephX keyring, selected through one audited checkpoint. Deny mode fails a request before any origin operation can touch the service credential.

reference today XrdPssConfig.cc: “We don't support credential forwarding, yet.”

docs/09-developer-guide/multi-user-backend-credentials-through-the-vfs.md

Owner-preserving deferred write-back

Detached write-back flushes to the origin as the original owner: the identity is persisted in a durable stage journal, re-resolved at flush time, and expiry-checked at time of use. In deny mode, a permanently lapsed credential dead-letters the flush — it never rides the service account instead.

reference today No equivalent: proxied writes carry the shared proxy identity.

src/fs/vfs/vfs_staged.c · stage journal

SigV4 → x509 credential minting

An S3 client authenticated by SigV4 can be issued short-lived x509 credentials for onward grid hops — bucket users participate in grid transfers without ever holding a certificate themselves.

reference today No S3 server surface exists, so no such bridge can.

docs/09-developer-guide/history-security-and-credentials.md

Per-request UNIX impersonation, red-teamed

Unprivileged workers apply per-request UNIX identity through a double-forked broker holding only CAP_SETUID/SETGID. Nine adversarial red-team rounds — 224 growing to 3,842 checks — without a boundary breach.

reference today Identity mapping exists; a capability-isolated, red-teamed broker does not.

docs/09-developer-guide/history-security-and-credentials.md

Identity-aware rate limiting

Limits and concurrency caps key on the authenticated principal — the mapped identity behind the token or proxy — so one hostile or runaway user throttles alone, and per-principal concurrency caps bound the blast radius.

reference today XrdThrottle keys on username only.

src/net/ratelimit/ · docs/07-security/threat-model.md

Macaroons: issue, not just verify

Full macaroon lifecycle — the server both issues caveated, expiring macaroons and verifies them, with expiry mandatory after an internal audit made it so.

reference today Macaroon support is verify-oriented; issuance depends on external services.

src/auth/token/ · history-security-and-credentials.md

Distinctive capability · protocols

More ways to reach the same data.

One namespace, with native S3, WebDAV class 2, GridFTP, and reporting interfaces available alongside root://.

A real S3 server endpoint

~6,900 lines of native SigV4 REST: GET, HEAD, PUT, DELETE, ListObjectsV2, multipart upload — served from the same tree as root:// and davs://. aws-cli and boto3 are first-class grid clients.

reference today S3 exists only as a client/backend; stock cannot serve S3.

src/protocols/s3/

WebDAV class 2

LOCK, UNLOCK, and PROPPATCH alongside the class-1 verbs and HTTP third-party-copy — enough for clients that demand real WebDAV semantics, not just GET/PUT over DAV headers.

reference today XrdHttp advertises DAV class 1: no locks, no property writes.

src/protocols/webdav/ · docs/04-protocols/webdav-directives.md

HTTP-TPC with OIDC token exchange

Third-party copy can delegate via OIDC with RFC 8693 token exchange — the token the remote end receives is exchanged and scoped for that transfer, not a pass-through of the user's bearer token.

reference today HTTP-TPC exists; RFC 8693 token-exchange delegation does not.

src/protocols/webdav/tpc_cred.c

A maintained GridFTP door

A gsiftp gateway with GSI control and data channels, MODE E parallel streams with restart markers, active and passive modes, and gsiftp↔gsiftp third-party copy — for ARC-CE fronts and legacy fabric that cannot migrate on someone else's schedule.

reference today The Globus toolkit is EOL, so sites that still need GridFTP must provide their own transition path.

src/protocols/gridftp/ · docs/05-operations/gridftp.md

WLCG Storage Resource Reporting, built in

SRR served natively over HTTPS/JSON — the accounting endpoint WLCG operations asks for, with no sidecar publisher, no cron job, no JSON template to hand-maintain.

reference today SRR publishing requires external tooling around the storage system.

docs/08-metrics-monitoring/ · SRR endpoint

Three protocols, one file, simultaneously

The same physical file is reachable as root://, davs://, and s3:// at once — one auth layer, one confinement rule, one cache, one metrics plane underneath all three. Not a gateway translating protocols: peers over one VFS.

reference today root:// and HTTP coexist; a unified S3-inclusive tri-protocol namespace does not.

docs/11-architecture/overview.md

Distinctive capability · integrity

Verify data at every stage.

Checksums everywhere stock has them — and in the places stock leaves blank.

CRC64 — both of them

crc64 (CRC-64/XZ) and crc64nvme calculators ship natively, alongside adler32, crc32c, md5, sha1, and sha256 — with the two 64-bit variants kept strictly distinct, encoded at the protocol edge.

reference today No CRC64 calculator ships with stock XRootD.

src/core/compat/checksum_core.c

Self-verifying writes

Every write path — POSIX, block store, S3; sequential or parallel-stream — funnels through one verified writer: a rolling, order-independent checksum accumulated as bytes arrive, folded with read-back verification at commit. Mismatch means unlink and a client-visible error, never a stored lie.

reference today Write-side self-verification of this shape has no stock equivalent.

src/fs/vfs/vfs_writer.c · src/core/compat/wverify.c

Verified reads from storage

CSI verification checks what storage hands back against the record made at write time — a bit-rotted block is an error you see, not a payload you serve.

reference today Read-path integrity depends on client-initiated checksum queries.

src/fs/backend/csi_verify.c

OCSP with stapling

Live revocation checking with stapled responses — revoked certificates are refused without every handshake paying an OCSP round-trip.

reference today CRL files only; no OCSP, no stapling.

src/auth/crypto/ocsp.c

Distinctive capability · client side

A complete client suite in the same package.

The server ships with its own complete, independently implemented client stack — no dependency on upstream libXrdCl or libXrdSec anywhere in it.

libxrdc and the tool belt

A C client library plus xrdcp, xrdfs, diagnostics (xrddiag), and checksum verification (xrdckverify) — built from the same in-tree protocol vocabulary as the server, so client and server can never drift apart silently.

reference today One client implementation exists: libXrdCl. There is no independent second.

client/ · client/apps/README.md

Three POSIX mount strategies

xrootdfs (FUSE, byte-exact under fault injection with zero EIO surfaced), an LD_PRELOAD shim for FUSE-averse hosts, and brixMount — a CVMFS-free writable union overlay for read-mostly software trees.

reference today A FUSE client exists; the preload shim and writable overlay do not.

client/apps/fs/ · client/lib/fs/overlay.c

Ceph migration tooling

The brix-tools package carries XrdCeph/CephFS migration utilities — the off-ramp for sites leaving a striped-RADOS deployment is in the same RPM set as the destination.

reference today Leaving XrdCeph is left as an exercise for the site.

packaging/rpm/ · brix-tools

A client-side metadata benchmark

metabench measures stat/open/list behavior of any endpoint from the client side — so capacity claims about your workload are measured, not modeled.

reference today No shipped metadata benchmark.

client/lib/observability/metabench/

Distinctive capability · operations

Clear information when you need it.

Observability and recovery behavior designed to help operators understand and resolve issues quickly.

Prometheus-native, dashboard included

Metrics are a first-class HTTP endpoint with fixed, low-cardinality labels, plus a built-in operational dashboard — scrape it with the stack you already run.

reference today Binary UDP f-stream/g-stream that needs a collector pipeline to become legible.

src/observability/

Origin-failure absorption

A stuck origin is detected in seconds (2 s connect / 4 s stall defaults), retried on fresh connections, and — when everything upstream is down — answered from cache with stale-if-error. A client storm coalesces into a single upstream fill.

reference today No stale-if-error; no request coalescing.

src/fs/backend/cache/ · deploy/cvmfs/

fail2ban, out of the box

Shipped filter and jail definitions use the sanitized access log to block scanners and repeated authentication attempts automatically, ready for deployment.

reference today No shipped fail2ban integration.

deploy/fail2ban/

An SELinux policy that says yes

An optional RPM subpackage labels /etc/grid-security for enforcing-mode hosts — the answer to the usual grid-service SELinux advice, which is to turn SELinux off.

reference today No shipped SELinux policy module.

packaging/rpm/nginx-mod-brix-cache.spec

A Kubernetes lab in the tree

Helm charts, in-cluster image builds with no external registry, and a remote suite that runs the full ~390-file test set from a client pod against a server pod — the cloud-native deployment story is tested CI infrastructure, not a blog post.

reference today Community charts exist; a tested in-tree k8s lab does not.

The BriX tool suite

Your muscle memory
still works.

BriX-Cache ships a complete client toolset in plain C — and the stock XRootD tool names keep working: xrdcp, xrdfs, xrdadler32, xrdgsiproxy. No XRootD client stack to install, nothing to relearn.

stock names, no stock install
$ xrdcp root://host//data/higgs.root .
$ xrdadler32 higgs.root
2f0a37b1 higgs.root
# each stock name is a personality of one small
# pure-C multi-call binary — symlinks, not a 400 MB stack
Why these tools

Four reasons to stop installing the big stack.

drop-in

The names you already type

Stock tool names stay invocable — xrdcksum answers as xrdcrc32c, xrdcrc64, xrdadler32, xrdckverify, and xrdcinfo; xrddiag absorbed xrdqstats, wait41, and mpxstats. Your scripts don't change.

self-contained

Small, plain C, hardened

Every tool is a thin front-end over one client library — pure C, no libXrdCl, no dependency tree. Binaries link with full RELRO, immediate binding, and non-executable stacks by default.

protocols

One toolset, every protocol

The same xrdcp that speaks root:// and roots:// also copies from and to davs://, http(s)://, and s3:// URLs, local paths, and -. One tool, one syntax, the whole storage estate.

resilient

Hardened for bad networks

Every tool rides the same machinery as brixMount: reconnect, full re-authentication, handle reopen, and offset resume inside one --max-stall patience budget. A flaky link costs you time, not a transfer.

The suite

Six families, one library underneath.

Data movement

xrdcp · xrdfs · xrd

xrdcp copies between root://, web URLs (davs, http(s), s3), local paths, and stdin/stdout — recursive, ZIP-member aware. xrdfs covers the filesystem verbs and doubles as an interactive shell. xrd is the swiss-army front-end: ls, stat, cat, cp, du, df, tree, find, locate, query, prepare — plus doctor, battery, and clockskew self-checks.

Checksums & verification

xrdcrc32c · xrdcrc64 · xrdadler32 · xrdckverify · xrdcinfo

One multi-call binary, five personalities. Hash a local or root:// file, verify a file on disk against its recorded checksum, or dump a cache file’s present-bitmap as JSON.

Diagnostics & monitoring

xrddiag · xrdqstats · mpxstats · xrdmapc · wait41

xrddiag turns raw failures into explanations — check, bench, watch, topology, compare, doctor. Query server stats and config, map a redirector’s live cluster, or block until a server accepts connections in your orchestration scripts.

Storage admin

xrdstorascan

verify pulls the bytes, recomputes the checksum, and compares against the server’s answer — end-to-end integrity, any backend. bench sweeps block size × parallelism and reports throughput, IOPS, and latency.

Auth & security

xrdgsiproxy · xrdgsitest · xrdsssadmin

Create, inspect, and destroy X.509 GSI proxies; self-test a GSI handshake against a live server; manage SSS shared-secret keytabs.

Staging

xrdprep

Issue kXR_prepare requests — stage, cancel, evict — for one or many paths on tape-backed or tiered storage.

Mounts & the POSIX shim

When the tool is “just a directory”.

brixMount

cvmfs · cvmfs-rw · eos · root · roots

The umbrella FUSE front-end: any CVMFS repository or XRootD/EOS endpoint as a local directory, torture-tested for bad networks. Read the brixMount page →

xrootdfs

one binary, two drivers

The XRootD FUSE mount itself: the async, resilient driver by default, and a simple synchronous fallback behind --legacy when you want the dumbest possible thing.

LD_PRELOAD shim

libbrixposix_preload.so

A POSIX→XRootD read-path shim for binaries you can't change: preload it and existing applications read root:// paths as if they were local files — no mount, no recompile.

A day with these tools

Three moments where it pays off.

The transfer that died at 90%

A 60 GB pull over a link that drops every few minutes.

$ xrdcp root://dc//run7/big.root .
# 54.1 GB in — connection reset by peer
# reconnect → re-auth → reopen → resume at 54.1 GB
[60.0/60.0 GB] done — checksum ok

A flaky link costs you time, not the transfer.

The error that explains itself

Auth fails five minutes before your slot on the batch farm.

$ xrddiag check root://dc.example.org
auth: GSI handshake failed
cause: proxy certificate expired 3h ago
fix: xrdgsiproxy init

The doctor is built in — causes and fixes, not error codes.

Prove the bytes, end to end

After a migration, “it copied” is not the same as “it’s intact”.

$ xrdstorascan verify root://dc//run7/big.root
local crc32c 8f3d21aa
server crc32c 8f3d21aa MATCH

Trust is a command, not a hope.

Bring your own tools

Already have a toolchain? Keep it.

The server side of the same promise: BriX-Cache speaks the wire protocols your existing clients expect, and the test suite runs the same conformance tests against BriX-Cache and the reference XRootD implementation.

root://

stock xrdcp · xrdfs · pyxrootd

The reference XRootD clients work unmodified — same opcodes, same semantics.

davs://

curl · rucio · browser

Standard WebDAV over HTTPS with x509 or WLCG tokens — anything that speaks HTTP works.

s3://

aws-cli · boto3

SigV4 REST against the same tree — point your existing S3 tooling at it.
